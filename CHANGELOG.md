## [Unreleased]

### Fixed

- **Chat input clears when switching chats in agent-playground.** `ChatInput` owned its own `value` / `images` `$state` that only cleared on submit, and `UserChat` is reused across chat switches (same route, different `chatId` prop) — so unsent drafts and attached images leaked between chats. Wrapped `<ChatInput>` in `{#key chatId}` so Svelte remounts it on every switch, resetting all of its local state. (#287, fixes #222)
- **Dropped image previews land above the chat input instead of at the top of the chat.** Dragging an image onto the chat surface wrote files into a separate `pendingImages` bucket in `user-chat.svelte` that rendered as a `.pending-images-bar` at the top of the component, while clicking the attach button used `chat-input.svelte`'s local `images` `$state` and rendered inside the input wrapper — same data, two state buckets, two render locations. Promoted `images` to a `$bindable()` prop on `ChatInput`, bound it from `UserChat` as `inputImages`, and routed the chat-surface drop handler at the same bucket so the file picker and the drop target share one preview strip. Also clears `inputImages` in the `chatId`-change `$effect` alongside the other per-chat resets, fixing a latent leak where dragged-but-unsent images followed the user into the next chat. (#294)

## [0.1.7] - 2026-05-12

Delegate sub-agent answers stop double-decoding through artifact pre-return lift (#271); the new `@friday/using-elicitations` skill documents the HITL surface job authors have had to discover by reading source (#275); a static migration manifest unblocks `deno compile` binaries that were silently enumerating zero migrations (#277). Cleanup pass on stale `FRIDAY_LOCAL_ONLY` references missed by #265 (#274).

### Added

- **Pre-return answer lift for delegate sub-agents.** `createDelegateTool`'s `execute` in `packages/core/src/delegate/index.ts` now routes every successful `answer` (whether emitted as plain text or via legacy `finish({ok:true, answer})`) through the new shared `liftAnswerForModel` walker in `packages/core/src/artifacts/scrubber.ts` before returning. Answers under the existing 8 KiB `TEXT_THRESHOLD_CHARS` pass through verbatim; over-threshold answers are uploaded as a `text/csv` / `text/markdown` / etc. artifact (MIME sniffed via `sniffTextMime`) and the returned `answer` becomes the `[attachment lifted to artifact …]` `REF_MARKER_PREFIX` marker text. Persistence's existing `REF_MARKER_PREFIX` short-circuit then skips re-lifting at the persist boundary, so a single artifact represents the answer end-to-end. The supervisor's next step references the artifact via `display_artifact` instead of re-emitting the prose it already saw stream, cutting the double-decode cost on document-shaped sub-agent answers (long emails, multi-row tables, document summaries, batched fetch results). (#271)
- **`liftAnswer` flag on `DelegateDeps`.** New `liftAnswer?: boolean` field on the `DelegateDeps` interface (default `true`) toggles the pre-return answer lift. Direct-execute callers that parse the answer text themselves opt out with `liftAnswer: false` — needed because the marker string is opaque to a downstream verdict parser. `packages/system/agents/judge-agent/judge-agent.ts` sets it on the judge's delegate deps so `parseVerdict` keeps seeing the raw verdict string. JSDoc on the field warns that any code path feeding the answer into another LLM should leave the default on. (#271)
- **`liftAnswerForModel` export from `@atlas/core/artifacts`.** New public function in `packages/core/src/artifacts/scrubber.ts` (re-exported from `packages/core/src/artifacts/mod.ts`) wraps the upload-and-marker projection as `(answer, { workspaceId, chatId, logger, serverId, toolName }) => Promise<{ value, artifactId? }>`. Short-circuits on the `REF_MARKER_PREFIX` (no double-lift on a re-emit) and on under-threshold inputs, never throws — upload failures log a warn and return `{ value: answer }` so a degraded artifact backend falls back to the original prose rather than dropping the delegate result. (#271)
- **`@friday/using-elicitations` system skill teaching FSM actions and Python `type: user` agents how to use the platform HITL surface.** New `packages/system/skills/using-elicitations/SKILL.md` (201 lines) documents the two MCP-registered elicitation tools — `request_tool_access(toolName, reason)` for allowlist approvals (fixed `Allow once / Allow always / Deny` options) and `request_human_input(question, options?)` for HITL decisions, approvals, confirmations, and disambiguation — both of which block the running tool call until terminal and return the answer envelope inline. Job authors already had these tools wired through MCP; they were undocumented anywhere centrally and the contract was discoverable only by reading `tools/agent-playground/src/lib/human-input/nested-choice.test.ts` and the MCP factory source. (#275)
- **Skill covers the nested per-item choice grouping shape the playground UI parses.** Documents the exact `[N] action — title` label / `N:action` value pattern that triggers the per-item choice-set renderer, the four detail-line keys the parser lifts into per-item headers (`Subject:` / `From:` / `Date:` / `Preview:`), the grouping constraints (at least two items, at least two actions per item, every option's value of the form `N:action` where `N` matches an item index in the question and every label starts with `[N]`), and the fall-back-to-flat-radio behavior when any constraint fails. Answer arrives in `answer.value` as a JSON-array string (e.g. `"[\"1:archive\",\"2:keep\"]"`) that the action must `JSON.parse` before dispatching; optional per-item free-text comments arrive in `answer.note` as `[N] free text` and are explicitly flagged as advisory rather than a parallel command stream. (#275)
- **Skill documents the blocking return envelope and status branches for `request_human_input`.** Return shape pinned as `{ ok: boolean, status: "answered" | "declined" | "expired", elicitationId: string, answer?: string, note?: string }` with explicit handling guidance — branch on `answer` for `"answered"` (parse first for grouped prompts), halt the side effect and summarize what was skipped on `"declined"` / `"expired"` without faking a value. Python `type: user` agents call the same primitive through `ctx.tools.call("request_human_input", {...})` with the identical return shape. (#275)
- **Skill documents the `request_tool_access` outcome matrix and authoring rules.** Seven-row `reason` → `granted` table covering `bypass` (workspace/daemon `dangerouslySkipAllowlist` proceed-silently), `persistent_allow` (prior Allow-always), `answered` (user picked one of the three buttons), `pending_user_approval` (created but unanswered — rare), `declined`, `expired`, and `unknown_tool` (not in the runtime catalog — stop guessing, do not retry). Spells out that the `reason` field is shown to the user verbatim so it must be moment-specific ("Need fs_write_file to save the patch we just drafted"), that `list_capabilities` / `list_mcp_tools` is the right pre-flight when the tool name is uncertain, and that `request_tool_access` should only be declared on actions that can produce a useful denial outcome rather than as a generic escape hatch. (#275)
- **Skill documents elicitation lifecycle, TTL inheritance, and retry idempotency.** TTL inherits from the parent job's `config.timeout` (default 30 minutes; per-job override via `jobs.<name>.elicitations.timeout`); the platform reuses any still-pending elicitation matching the same `(question, options, sessionId, actionId)` tuple rather than creating a duplicate so accidental LLM retries are idempotent; expired prompts move to a read-only Activity log entry that cannot be answered retroactively, and acting on a timed-out prompt does not reify the underlying job. (#275)
- **Skill enumerates the high-traffic authoring gotchas.** Plain-text-only `question` rendering (markdown tokens become literal `**` / `#` / `- ` / fenced-code characters in the UI), no schema support for `multi_select` / `default` / `required` / nested option objects (item grouping must be encoded into label/value strings), grouping fallback when fewer than two items or fewer than two actions per item, the `request_tool_access` vs `request_human_input` mix-up where "which provider" is a `request_human_input` question with `[Gmail, Outlook]` options not a tool-access request, the dead-end where an `outputTo` LLM action prints a prose menu and tries to wait for a future user message (the run ends without the answer because the action only resumes through the elicitation envelope), and the advisory-only nature of `answer.note` content. (#275)
- **`apps/atlasd/src/migrations/index.test.ts` manifest-membership guard so a forgotten registration trips CI.** Adds three sibling assertions that read `index.ts` as text (importing it would pull `@db/sqlite` transitively, which the existing tests in the file already avoid): every `m_*.ts` on disk has a `from "./<file>"` import line AND a `\bm_<id>\b` token inside the sliced `MIGRATIONS` array body, and the array's non-comment entry count matches `listMigrationFiles().length` exactly so deleting a file but forgetting to remove its array entry (or vice versa) fails the size assertion. The slice is computed by a factored-out `extractMigrationsArrayBody(indexSource)` helper — regex `/const MIGRATIONS:\s*readonly Migration\[\]\s*=\s*\[([\s\S]*?)\];/` followed by `.replace(/\/\/.*$/gm, "")` — so a bare `m_<id>` reference in a doc comment elsewhere in the file, or a temporarily `//`-commented array entry, cannot satisfy a whole-file scan and silently let a missing registration through. The size check filters lines via `/^m_\d{8}_\d{6}_[a-z][a-z0-9_]*$/` after trimming trailing commas. A dedicated `describe("extractMigrationsArrayBody")` block locks in the predicate with three cases (slice between brackets, comment-only entry rejected, null when the array literal is absent) so a future refactor of the assertion shape can't quietly weaken the guard. (#277)

### Changed

- **Delegate sub-agent system prompt: text-on-success contract.** The child-prompt template at `createDelegateTool` (`packages/core/src/delegate/index.ts`) now directs the sub-agent to emit final answers as plain text content — "your text reply IS the answer" — and to call the `finish` tool ONLY on failure with `{ ok: false, reason }`. Removes the wasted double-decode where the model wrote the same long answer once as text and once as a `finish` argument. The instruction against "markdown tables, section headers, or other human-facing formatting" was dropped from the terseness directive so the child can still produce structured document-shaped output (the whole point of the lift path). The `finishTool` description in `packages/core/src/delegate/finish-tool.ts` was rewritten to match: "ONLY when the task is impossible to complete"; the `{ok: true, answer}` variant remains for back-compat but is no longer the recommended success path. (#271)
- **Empty delegate output now surfaces as a structured failure.** Both the `finish({ok:true, answer})` branch and the text-on-success branch in `createDelegateTool.execute` now `.trim()`-check the candidate answer and, when it's empty/whitespace-only, return `{ ok: false, reason: "delegate produced no answer", …resultBase }` instead of the previous silent `{ ok: true, answer: "" }`. Under the new text-on-success contract a malformed step that completes cleanly with no output is otherwise indistinguishable from real success; the supervisor now sees the structured failure and can act on it. (#271)
- **`tool-suite-management` promptfoo suite surfaces the real workspace-chat prompt.** `tools/qa/live-daemon/promptfoo/tool-suite-management-config.yaml` now loads `packages/system/agents/workspace-chat/prompt.txt` as `vars.system_prompt` and `prompts:` points at a new shared `tools/qa/live-daemon/promptfoo/prompt-template.txt` that renders the system prompt + per-test `user_message` side-by-side — mirroring the `tool-choice` suite. The six scenarios in `tool-suite-management-tests.yaml` now declare each `user_message` literally (the actual text `postChatMessage` forwards to the chat endpoint). The `promptfoo view` prompt panel now shows what the model saw rather than just the `scenarioId` placeholder; other daemon-level suites (`retrieval-gated-config.yaml`, `table-outcomes-config.yaml`, `delegate-skills-config.yaml`, `load-skill-injection-config.yaml`) keep the `{{scenarioId}}` placeholder because they exercise daemon plumbing rather than a single LLM prompt. All 62 promptfoo cases still pass. (#271)

### Fixed

- **Migration manifest replaced with static imports so `deno compile` daemons actually run migrations.** `apps/atlasd/src/migrations/index.ts`'s `getAllMigrations()` previously walked the directory with `readdir(fileURLToPath(new URL(".", import.meta.url)), { withFileTypes: true })` and pulled each `m_*.ts` file in via `await import(new URL(file, dir).href)`. Inside a `deno compile` binary `import.meta.url` resolves into Deno's compile-internal virtual FS that `node:fs/promises#readdir` returns nothing for, and dynamic-import-by-URL isn't wired into the bundled module graph — so shipped daemons silently enumerated zero migrations, `friday migrate` reported `ran 0 migration(s); skipped 0 already applied`, and downstream bootstrap (notably `m_20260511_110800_provision_workspace_members` stamping owner rows into `WORKSPACE_MEMBERS`) never executed, leaving every workspace-scoped HTTP route to 403 because the membership row was missing. The new module hard-codes 22 named imports (one `import { migration as m_<id> } from "./m_<id>.ts"` per file from `m_20260501_120000_chat_to_jetstream` through `m_20260511_120000_cleanup_userid_as_name`) into a `const MIGRATIONS: readonly Migration[]` literal; `getAllMigrations()` is now a synchronous spread + `[...MIGRATIONS].sort((a, b) => a.id.localeCompare(b.id))` returned as `Promise.resolve(...)` for back-compat with the daemon's and CLI's existing `await` callers. Verified by `deno task atlas migrate --dry-run --json` against the compiled binary enumerating all 22 migrations in `skipped` (vs. 0 on `main`). (#277)
- **Block-comment attribution corrected: defensive sort lives in `getAllMigrations`, not `runMigrations`.** The new manifest's leading doc comment originally claimed that `runMigrations` re-sorts the array defensively, but `packages/jetstream/src/migrations.ts` iterates the input array in input order — the only sort is the `localeCompare` inside `getAllMigrations()`. The comment now correctly attributes the defensive sort so future maintainers don't assume out-of-order entries in `MIGRATIONS` are harmless at the consumer end. (#277)
- **`<details>` cards in `tool-burst.svelte` and `delegate-tool-card.svelte` preserve the user's manual open/closed state across re-renders.** Both components used `<details open={initialOpen}>` with `initialOpen` snapshotted once via `untrack`, but Svelte 5 re-applies the bound `open` attribute on every parent re-render of the surrounding reactive scope — so any downstream re-render (post-elicitation cache update, SSE push, virtualizer measurement) wrote `open={false}` back over a user-driven `open=true` from a summary click. Symptom: an already-answered elicitation card could not stay expanded — clicking the burst summary opened it momentarily, then the next re-render snapped it shut. Switched both components to `bind:open={isOpen}` with `isOpen` declared as `$state(untrack(() => ...))` initialized from the same snapshot, so the user's click and the live-only force-open `$effect` (which now writes to the state, not directly to `detailsEl.open`) are the two sources of truth; the `bind:this={detailsEl}` handle is removed entirely. Plain `open={state}` one-way binding has the same re-apply problem — `bind:open` is the only stable shape. (#275)

### Configuration

- **`docker/run-platform.sh` swaps `FRIDAY_LOCAL_ONLY` for `FRIDAY_ENV=dev`.** `FRIDAY_LOCAL_ONLY` was renamed to `FRIDAY_ENV` in #265, but `docker/run-platform.sh` was missed in that sweep and still exported the dead var, which left the containerized daemon in the new fail-closed posture — every `/api/*` request hit the `apps/atlasd/src/session-middleware.ts` 401 path because no session-minting gate was tripped. The launcher now exports `FRIDAY_ENV="${FRIDAY_ENV:-dev}"` so the daemon auto-mints a local session and skips remote credential fetching, mirroring what the Studio installer (`apps/studio-installer/src-tauri/src/commands/env_file.rs`) and `atlas daemon start` already do. (#274)
- **`.env.example` documents `FRIDAY_ENV` with the new fail-closed semantics.** The commented-out `FRIDAY_LOCAL_ONLY=0` example (whose "disable outbound calls" copy no longer matched the var's behavior even before the rename) is replaced with `# FRIDAY_ENV=dev` plus a description block spelling out that `dev` auto-mints a local `/api/*` session and skips remote credential fetching, while any other value — including unset — requires a valid `friday_session` cookie or Bearer token and returns 401. Notes that Friday Studio and `atlas daemon start` set this automatically, so the example matters only for operators with hand-rolled `.env` files. (#274)

### Removed

- **Placeholder-JWT `FRIDAY_KEY` auto-generation deleted from `docker/run-platform.sh`.** The 13-line `node -e` block that minted an unsigned HS256 JWT with `iss: 'friday-platform'`, `email: 'platform-local@hellofriday.ai'`, `sub: 'local-user'`, and a `.local` signature suffix is gone, along with the `[platform] Auto-generated FRIDAY_KEY for local user identity` log line. The daemon stopped decoding `FRIDAY_KEY` for user identity in #265 (that JWT-decode path and the literal `"local-user"` sentinel were removed in favor of `UserStorage.resolveLocalUserId()`), and an unset `FRIDAY_KEY` already disables the upstream gateway path in the CLI, so the fake JWT served no purpose except masking that the var was unconfigured. Net change: `docker/run-platform.sh` drops from 30 lines of identity bootstrap to 4 lines of dev-mode setup. (#274)

## [0.1.6] - 2026-05-12

Workspace isolation + per-user authz (#265) and the tool-suite + prompt-caching redesign (#261) are the headline; HTML chat exports (#198), nested-usage accounting across NATS (#262), and OAuth refresh resilience (#267) round out the cycle. Read **Breaking changes** before upgrading.

### Breaking changes

- **`FRIDAY_LOCAL_ONLY` renamed to `FRIDAY_ENV`; daemon fails closed when unset.** Single-user identity used to be inferred from the literal `local-user` baked into a placeholder JWT; that surface is gone. The daemon now authenticates `/api/*` via an opaque `friday_session` cookie (or `Authorization: Bearer`) validated against a new `SESSIONS` JetStream KV bucket, and a global Hono middleware in `apps/atlasd/src/session-middleware.ts` stamps `ctx.userId`. Only `FRIDAY_ENV=dev` auto-mints a session bound to the canonical `UserStorage.resolveLocalUserId()` nanoid; any other value — including unset — returns 401. The Studio installer (`apps/studio-installer/src-tauri/src/commands/env_file.rs`) and `atlas daemon start` both write/inherit `FRIDAY_ENV=dev` so local-first usage is unaffected; operators with hand-rolled `.env` files who relied on the implicit `FRIDAY_LOCAL_ONLY` default must add the explicit value. `FRIDAY_KEY` remains as an outbound Bearer credential for Link / persona / share / OTEL ingest — orthogonal to who-is-the-user. The local placeholder JWT, the adapter's JWT-decode path, the userId cache, and `apps/atlas-cli/src/services/local-friday-key.ts` are deleted. (#265)
- **Admin-only surfaces 403 outside `FRIDAY_ENV=dev`.** `/api/chat-storage/*` (in-memory conversation debug list/get/replace with no per-tenant scope), `/api/config/env` (daemon `.env` read/write), `/api/link/*` (shared-credential proxy that authenticates to Link with the daemon's own `FRIDAY_KEY`), and the `bundle-all?include=global-skills` operator escape hatch are now gated by a new `apps/atlasd/src/dev-only.ts` middleware that throws `HTTPException(403)` whenever `FRIDAY_ENV` isn't `dev`. Local Studio / CLI flows are untouched because both stamp `FRIDAY_ENV=dev` automatically. (#265)
- **Per-feed live-event SSE endpoints removed.** `GET /api/elicitations/stream` (workspace-scoped), `GET /api/elicitations/stream/global` (sanitized), and the `?stream=true` branches on `GET /api/instance/events` and `GET /api/events` are deleted; every playground consumer now subscribes through the per-user firehose at `GET /api/me/stream`. Replay endpoints (`/api/instance/events`, `/api/events` with `since=…`) and `/api/sessions/:id/stream` are retained — the first two as first-load hydration paths for the SharedWorker, the third because its JetStream pull-consumer shape doesn't fit the firehose yet. External clients that wired against the per-feed `?stream=true` endpoints must migrate to the user firehose or hold on the replay endpoints. The unused `ElicitationSummarySchema` / `ElicitationSummary` exports are dropped from `@atlas/core`. (#265)
- **Platform tool surface renamed verb-first across both chat and MCP server.** No compat aliases — old names are rejected. `memory_save` → `save_memory_entry`, `memory_read` → `list_memory_entries` (rich filter shape with `query/since/until/metadata/limit/cursor/order/truncate`), `memory_remove` → `delete_memory_entry`, `artifacts_create` → `create_artifact`, `artifacts_get` → `get_artifact`, `workspace_delete` → `delete_workspace`, the polymorphic `remove_item({kind, id})` is split into `delete_agent` / `delete_signal` / `delete_job`, and `get_mcp_dependencies` is subsumed by `describe_mcp_server({id})`. Lift-marker text in `scrubber.ts` and every `prompt.txt` callsite is updated; the bumped `friday-agent-sdk` 0.1.8 picks up the matching `ctx.tools` renames. Installed Python agents that hardcode old names need the repair recipe in `@friday/repairing-workspaces-agents`. (#261)
- **Auto-injected `<jobs>` and `<integrations>` blocks dropped from system block 2.** Each workspace job is still bound as a callable tool with the same name, but discovery now happens via the new `list_jobs` / `describe_job` and `list_integrations` / `describe_integration` retrieval tools. Foreground workspaces ride as a names-only `<foreground_workspaces>id1, id2</foreground_workspaces>` tag instead of inlining each foreground's full `<workspace>` block — the chat calls `describe_workspace(id)` on demand. Cuts block 2 bytes for the multi-foreground case and lets description-only edits to skills stop busting the prefix cache. (#261)
- **`<persona>` and `<tabular_data>` blocks removed from `prompt.txt`.** ~50 lines of seven-persona frame guidance and the "UI can't render markdown tables, use bullet lists" rule are gone — chat now renders markdown tables natively (with horizontal scroll + Actions dropdown) and persona inferences live in user-authored memory, not the system prompt. (#261)

### Migration

- **`USERS` JetStream KV bucket provisioned + backfilled from legacy memory identity entries.** Identity moves out of the workspace-scoped `notes` narrative (type:user-name / type:name-declined entries) into a cross-workspace `User { identity, preferences, onboarding }` record keyed by a 12-char alphanumeric nanoid. `m_20260504_025000_provision_users_bucket` reads the personal workspace's `notes` for legacy entries and writes them into the local user's USERS record; idempotent via `nameStatus !== "unknown"` OR `onboarding.version >= ONBOARDING_VERSION`, so a partial-crash rerun that wrote `setUserIdentity` but missed `markOnboardingComplete` no longer re-derives from legacy memory and overwrites a possibly-already-edited identity. Onboarding no longer re-asks when the user crosses workspaces. (#261)
- **`default-user` chat metadata re-keyed to the resolved local user nanoid.** `m_20260504_025500_rekey_default_user_chats` walks every `CHATS` KV entry, finds records still stamped with the literal `"default-user"` sentinel (the pre-Phase-0.5 fallback when no `FRIDAY_KEY` was configured), and CAS-rewrites the `userId` field — KV keys (`<workspaceId>/<chatId>`) and message stream contents are untouched. Collects all keys first to work around the NATS JS-KV consumer's interleaving issue (reads inside `kv.keys()` for-await silently drop later keys). Idempotent. (#261)
- **`sessions-v2 → JetStream` migration is safe to retry.** Each republished event line now carries a stable `Nats-Msg-Id = migrate-v2:<sid>:<idx>`, and the `SESSION_EVENTS` stream is created with a 24-hour `duplicate_window` (matching the runtime adapter's `DEFAULT_DUPLICATE_WINDOW_NS`). A crash partway through, or a rerun after a daemon restart that lands more than 2 minutes after the original run, no longer silently double-publishes lines into stream history. Born-with-24h on create; the runtime adapter already reconciles existing streams to the same window via `streams.update`. (#261)
- **`WORKSPACE_MEMBERS` JetStream KV provisioned and backfilled on first boot.** New per-workspace membership bucket keyed `<userId>.<wsId>` with values `{role, addedAt}` (`role ∈ owner | admin | member | agent`); see `packages/core/src/workspace-members/`. The `apps/atlasd/src/migrations/m_20260511_110800_provision_workspace_members.ts` migration walks `WORKSPACE_REGISTRY`, derives the owner from `metadata.createdBy` (falling back to the resolved local user nanoid for legacy rows), and stamps one `owner` row per workspace via `putIfAbsent` so it's idempotent. Steady-state writes happen in `WorkspaceManager` via the new injected `MembershipWriter` — the manager stays storage-agnostic, the daemon wires it against `WorkspaceMemberStorage` at boot. (#265)
- **`USERS` bucket cleanup migration reverses the userId-as-name placeholder bug.** Before, `/api/me`'s local adapter substituted `full_name = <userId>` and `email = <userId>@local.friday` for unset fields (the response schema was non-nullable strings); `workspace-chat/user-identity.ts` then auto-synced that placeholder back into the USERS record with `nameStatus: "provided"` and stamped `onboarding.completedAt`, so the system prompt's `<user_profile>` block surfaced the nanoid as the user's name forever and the welcome wizard never re-fired. `m_20260511_120000_cleanup_userid_as_name.ts` detects the precise tuple (`nameStatus === "provided"` AND `name === userId` AND `email === <userId>@local.friday`) and resets `identity.name` / `identity.email` to unset, flips `nameStatus` back to `"unknown"`, and resets `onboarding.version` to 0 so the wizard runs again on next playground visit. The `/api/me` adapter has been fixed to surface `null` for unset fields so the auto-sync no longer triggers in the first place. Other identity fields (`timezone`, `locale`, `declinedAt`, `preferences`, `createdAt`) are preserved; CAS conflicts on concurrent writes log-and-continue. (#265)

### Added

- **Workspace membership store with role-based authz across every workspace-scoped HTTP route.** Each `/api/workspaces/:workspaceId/*` handler now resolves the caller's membership before doing work: read/act endpoints call `requireWorkspaceMember` (any of `owner | admin | member | agent` passes), config edits + member-management + cron pause/resume narrow to `requireWorkspaceAdmin`, both 403 on a missing row and 401 when `ctx.userId` is absent. Listings filter to the caller's accessible set via a single `getAccessibleWorkspaceIds` prefix-scan — cron's all-timers listing, the workspaces index, sessions index, jobs index, skills, memory, mcp-registry, integrations, chat, chat-debug, communicator-connect, draft, create, export, bundle-all, etc. all gate through the new `apps/atlasd/src/workspace-authz.ts` helpers. Elicitation get/answer/decline resolve the workspaceId indirectly through the elicitation id, then enforce membership. (#265)
- **Per-user event firehose `GET /api/me/stream` with live-watched membership authz.** One long-lived SSE per browser multiplexes `elicitations.>`, `events.>` (workspace lifecycle / schedule-missed) and `instance.>` (cascade) into a discriminated-union frame shape (`{kind, workspaceId?, subject, payload}`), replacing the prior fanout of `tabs × feeds` EventSources that exhausted HTTP/1.1's per-origin connection budget. Authz is per-event: at handshake the handler opens `openAccessibleWorkspaceWatch(nc, userId)` which fuses a snapshot of `WORKSPACE_MEMBERS` with a `KvWatchInclude.UpdatesOnly` watch on the `<userId>.>` prefix, so a workspace shared mid-session propagates to the running stream without forcing a reconnect. Subscribe-then-flush so the handshake honestly means "from now on, every event;" 15s `: keepalive` heartbeats keep intermediate proxies from idle-closing. ### Fixed

- **Chat handler buffer-capture race.** The handler captured `ownBuffer = streamRegistry.getStream(chatId)` *after* awaiting `thread.subscribe` and `ChatStorage.appendMessage` — in that 10–50ms window, a follow-up POST could replace the chatId-keyed buffer, causing the handler to write turn 1's late chunks into turn 2's buffer. Now stashes the buffer on `Message.raw.turnBuffer` at dispatch and reads it back deterministically; non-web adapters (Slack / Telegram / Teams) that never create a buffer no longer log spurious `stream_event_dropped` warnings. (#206, follow-up to #192/#193)
- **Cross-turn event leak in chat stream registry.** Two cooperating bugs let a stopped turn's late events corrupt the next turn's stream: `finishStream(chatId)` was called in `finally` without identity verification, so a fresh buffer for the same chatId was flipped to inactive by the prior turn's cleanup; `appendEvent` looked up buffers by chatId only, so late events from an aborted turn (subprocess emit, NATS reconnect-buffer flush, streamSub drain) landed in whatever buffer was current. Now the handler captures `ownBuffer` once and threads it through both `appendEvent(chatId, chunk, ownBuffer)` and `finishStreamIfCurrent(chatId, ownBuffer)` so the identity check lives at the registry boundary. (#194)
- **`{{inputs.x}}` interpolation in agent action prompts.** Friday workspaces (which exclusively use agent actions) previously saw literal `{{inputs.description}}` in the prompt while the resolved values sat in an appended `## Input` block. - **MCP cold-start probe timeout on first install.** The 5-second foreground probe was too tight for `npx` / `uvx` MCP servers on first run — npm cold-downloaded the package in 5–6s, timing out the user's first click while the second succeeded against cached npm. - **FSM-path MCP servers now receive registry `platformEnv` variables.** The code-execution path through `runtime.ts` was handing raw workspace MCP server configs to the FSM engine without applying registry-owned `platformEnv` (`MCP_ENABLE_OAUTH21`, dummy `GOOGLE_OAUTH_CLIENT_ID` / `SECRET`, `WORKSPACE_MCP_STATELESS_MODE`). Mirrors the same pattern used in `executeCodeAgent`. (#196)
- **MCP subprocess registry evicts stale children on spec drift.** `sharedMCPProcesses.acquire` was first-spawner-wins: whichever caller hit `acquire(serverId, …)` first locked their `spec.env` for the daemon's lifetime, so a cached subprocess spawned by a buggy code path served stale config until restart. ## [0.1.1] - 2026-05-01

### Added

- Workspace-scoped skill assignment tools (`assign_workspace_skill` / `unassign_workspace_skill`) for the workspace-chat agent. Calls `SkillStorage` directly, validates `@namespace/name` refs via `SkillRefSchema`, and is idempotent. Assignments take effect on the next chat turn because `resolveVisibleSkills` reads the assignment table every turn. System prompt and `workspace-api` skill reference updated to distinguish ephemeral `load_skill` from persistent workspace assignment. (#114)

### Fixed

- Hung MCP servers no longer block workspace chats. `Promise.allSettled` replaces the serial connect loop so one stuck server cannot stall the others, and hard timeouts are enforced at every phase: 4s on the HTTP reachability probe, 20s on `createMCPClient` + `listTools` handshake, and a 15-minute ceiling on `callTool` invocation. Timed-out servers are silently dropped; structured `operation: "mcp_timeout"` telemetry (with `serverId`, `phase`, `durationMs`) is logged for future circuit-breaker decisions. The previous retry on hang was removed — one shot, one timeout, move on. (#116)
- Four macOS launcher / tray bugs (#113):
  - Autostart plist now points at the bundle ID the installer actually writes (`ai.hellofriday.studio`), not the never-installed `-launcher` suffix. LaunchServices no longer fails autostart and the tray "Restart" action with `LSCopyApplicationURLsForBundleIdentifier`. Existing broken installs self-heal on next launcher start.
  - Tray "Restart all" no longer paints the menubar `Error` for the few seconds children take to come back up. A new restart-grace window (active during the in-flight restart and for `bucketFailGraceWindow` after) suppresses the transient `AnyFailed` bucket.
  - Tray "Restart all" no longer terminates the supervisor watchdog. `RestartAll` now calls `processRunner.RestartProcess` per service in `startOrder`, which keeps `pendingRestarts > 0` and prevents process-compose's `runner.Run` loop from exiting "Project completed" between the stop and start passes. As a side benefit, four of five services stay running at any instant during a user-initiated restart.
  - The user's "Start at login" disabled choice now survives launcher restarts. `isAutostartStale()` on darwin matches the Windows contract: stale iff the registered value is both non-empty and mismatched. An absent plist means user-disabled and is left alone.
  - The cold-start grace and post-restart grace are unified behind a single `bucketFailGraceWindow` constant (90s) that covers the readiness-probe budget (`InitialDelay=2s + FailureThreshold=30 × PeriodSeconds=2s = 62s`), so legitimately slow first-pass readiness (the friday daemon's ~24s workspace scan + skill bundle hashing + cron registration on cold start) no longer flips the bucket red mid-startup.
