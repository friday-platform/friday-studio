# Changelog

All notable changes to this project will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [0.1.11] - 2026-05-20

A polish release. **macOS TCC denials** that the chat agent hits when reaching into `~/Downloads` / `~/Desktop` / `~/Documents` now surface as an in-card affordance with a Settings deeplink and a one-click `mv`-out suggestion, instead of dumping raw stderr at you (#406). The playground's **Run button** stops cancelling its own jobs when you navigate away — long cascades survive a refresh or tab close (#402). Voice input in the chat composer **keeps the mic listening** until you click it off, matching ChatGPT-style manual stop instead of cutting out at the first speech pause (#407). MCP installs of curated OAuth providers (Notion, Linear, Atlassian, PostHog) stop dead-ending on a misleading "Conflict enabling MCP server." (#400). The sidebar "Add Space" dialog now accepts `.zip` workspace bundles alongside loose YAML (#395), and the **`labels` LLM role drops its hardcoded Groq default** — environments with `GROQ_API_KEY` were silently routing session titles through Groq and 403ing in geo-blocked regions; default is now `anthropic:claude-haiku-4-5` (#412). Plus a sidebar refactor that collapses three near-identical tree implementations onto a single primitive (#397), and **Dependabot PRs no longer auto-merge** (#401).

### Added

- **macOS TCC denials surface as an in-chat card.** When `run_code` or `file_io` tool calls hit `Operation not permitted` against a TCC-protected root (`~/Downloads`, `~/Desktop`, `~/Documents`), a new `tcc-detect.ts` recognizes the three real-world stderr shapes (`find:`, `ls:`, Python `PermissionError`) and `code-exec.ts` / `file-io.ts` attach a structured `tcc_denied` payload to their envelopes. The renderer shows a yellow `tcc-denied-card.svelte` with a Settings deeplink (`x-apple.systempreferences:`) button and a clipboard-copy `mv` button; `tool-call-utils.ts` extends `needsUserAction` so `tool-burst`'s collapsed `<details>` auto-opens — previously the card rendered but stayed invisible. Gated on `process.platform === "darwin"` (same error on Linux/Windows usually indicates a genuine ACL issue) and only addresses the already-denied state; the first-encounter consent dialog is still handled by macOS natively. (#406)

### Changed

- **Unified `SidebarNav` primitive for skills, MCP catalog, and chat list.** Three near-identical hand-rolls (`skills-tree.svelte`, `mcp-catalog-tree.svelte` + `mcp-section-nav.svelte`, `chat-list-panel.svelte`) each re-implemented tree-line connectors, hover states, search pills, and active-row styling. A new compound `SidebarNav` (`Root` / `Search` / `Group` / `Item`) under `packages/ui/src/lib/sidebar-nav/` recursively renders `subItems` in place with the skills tree's L-branch connectors, supports `href` / `onclick` / `bare` triggers, ships a `variant="sub"` underline-active mode, and an `actions` snippet that escapes the trigger's `<button>` to host sibling controls without nested-button HTML. `mcp-section-nav.svelte` is deleted: MCP sections now render as `subItems` under the selected server, and re-clicking an open server collapses back to `/mcp`. Skills namespace groups default to open; the Skills "Add" trigger drops its icon to match MCP's "Add New". (#397)
- **`labels` LLM role default is anthropic-only; Groq is opt-in via `models.labels`.** The default chain in `packages/llm/src/platform-models.ts` previously led with `groq:openai/gpt-oss-120b`, so any environment with `GROQ_API_KEY` set silently routed session titles (and every other `smallLLM` caller) through Groq — including in regions where Groq is geo-blocked, where the call surfaced as a 403 on every session title. Default is now `anthropic:claude-haiku-4-5` across all four roles; opting back into Groq for sub-second titles is a one-line addition: `models.labels: "groq:openai/gpt-oss-120b"` in `friday.yml`. User-configured `models.labels` was always honored and is unchanged. (#412)
- **Dependabot PRs no longer auto-merge on green CI.** `.github/workflows/dependabot-auto-merge.yml` is deleted. The workflow listened on `pull_request_target` and ran `gh pr merge --auto --squash` against any PR authored by `dependabot[bot]` — because `main` is not branch-protected, that effectively landed bumps the moment required checks went green, with no human in the loop. Bumps now stay open until someone manually merges them. (#401)

### Fixed

- **Run button no longer cancels its own jobs on browser navigation.** `run-job-dialog.svelte` flips from sync `POST /api/workspaces/<id>/signals/<sig>` to `?nowait=true`. The previous sync POST kept the inbound HTTP socket open for the entire cascade — the modal closed immediately so it *looked* fire-and-forget, but the fetch was alive in the background; any navigation, refresh, or tab close aborted the request, the daemon treated that as `Client disconnected`, and cancelled the spawned session. With `?nowait=true` the daemon publishes to JetStream and returns `202 {status:"accepted", correlationId}` in <100 ms, decoupling the cascade from the browser tab's lifetime. To preserve the land-on-the-session UX, the dialog snapshots existing `sessionIds` for this job *before* the POST and polls `client.sessions.index.$get` for the first new session whose `jobName === jobId` (~5.5s budget); snapshot-then-diff avoids clock-skew misses that a `startedAt >= clickTs` compare would hit. A component-scope `AbortController` threads through trigger and poll so a slow poll can't `goto()` after Cancel/unmount. (#402)
- **MCP doctor skipped for curated providers; real enable errors surfaced to the agent.** `apps/atlasd/routes/mcp-registry.ts` used `!translateResult.linkProvider` as a proxy for "no auth info available," but the translator deliberately suppresses `linkProvider` when `REGISTRY_ANNOTATIONS` supplies a curated `providerId` — so the doctor ran on fully-specified HTTP-remote OAuth servers (Notion, Linear, Atlassian, PostHog), returned verdict `unknown`, and cascaded into a `needs_manual_config` 409. Fix adds a `hasCuratedProvider` clause to the `needsDoctor` predicate so annotated entries take the synchronous README-fetch fast path. In parallel, `enable-mcp-server.ts` only read `errorBody.message`, collapsing the diagnostic `needs_manual_config` shape (which returns `error` + `serverId` without `message`) into the opaque string "Conflict enabling MCP server."; the 404 / 409 / 422 branches now fall back to `errorBody.error`, and `res.json()` is wrapped so gateway 502 / 504 HTML bodies no longer escape as parse errors. (#400)
- **Playground voice-input mic stays live until the user stops it.** The Web Speech API integration in `chat-input.svelte` constructed `SpeechRecognition` with `continuous = false`, ending the session on the first detected speech pause and aborting the recording before the user was done talking. Flipped to `continuous = true`, so the mic records until the user explicitly clicks the toggle again. `interimResults = true` and the `navigator.language` locale binding are unchanged, so interim transcripts still stream into the textarea. Known limitation: Chrome enforces an internal ~60 s cap even with `continuous = true`; a restart-on-`end` handler is a follow-up. (#407)
- **`studio-release publish` no longer stalls on a `less` `(END)` prompt mid-flow.** `scripts/studio-release.sh` calls `gh run view` to print a workflow summary, but `gh` auto-pipes through `$PAGER` whenever stdout is a TTY — the publish flow froze on `(END)` and required a manual `q` keystroke before continuing. Prefixed that single invocation with `GH_PAGER=cat`. (#399)
- **Sidebar "Add Space" dialog accepts workspace `.zip` bundles, not just YAML.** The dialog's file picker hardcoded `accept=".yml,.yaml"`, so users with an exported workspace zip had to go to Settings → Import a workspace instead. `workspace-loader.svelte` now branches on file type: YAML files still parse inline and `POST /api/workspaces/create`; `.zip` files route through `POST /api/daemon/api/workspaces/import-bundle` as a multipart upload, mirroring the Settings flow. Both `handleDrop()` and `handleFileInput()` validate by extension + MIME (Safari/macOS send `application/x-zip-compressed`) so dropping a `.txt` fails with a clear message instead of a downstream YAML parse error. Bulk-export zips (multiple workspaces per archive) remain Settings-only. (#395)
- **Job, edit, and memory pages share one breadcrumb implementation.** Three playground routes had drifted onto different breadcrumbs — job/edit used the bespoke `WorkspaceBreadcrumb` (back-arrow + workspace dot), memory hand-rolled inline `<nav class="breadcrumb">` tags. The memory page also lacked a `PageLayout.Root` wrapper, so its crumbs escaped to the viewport edge. All three now render through `<PageLayout.Root>` → `<PageLayout.Breadcrumbs {crumbs} />` → `<PageLayout.Body>`, matching `/agents`, `/settings`, `/schedules`. (#411)


## [0.1.10] - 2026-05-19

A reliability and polish release. **Microsoft Mail** is now a connector — link your personal Outlook or work Microsoft 365 mailbox and Friday can read, search, and send mail on your behalf (#389). On macOS, **Friday Studio comes back on its own if it crashes** instead of leaving you with a missing tray icon until the next login (#345). The chat sidebar no longer gets stuck on **"Offline" after long sessions** — it recovers on its own where you previously needed to restart the app (#369). When agents need to set a secret like an API key, you can now **type it straight into the confirmation card** so the value never shows up in your chat history (#352). Installed builds also log a lot less by default, so daemon logs stay readable (#392). Plus a design pass that brings consistent typography, spacing, and layout across the playground (#387).

### Added

- **`env_set` confirmation card keeps secret values out of chat history.** Agents propose env writes through an in-chat card; the user reviews, edits, and confirms. For secret-looking keys (`token`, `key`, `secret`, `password`, `credential`), the agent leaves the value blank and the user types it into a password input. Exception: if the user pasted the literal value in chat first, the agent passes it through. After confirmation the card mirrors what's in `.env`, so edits survive a page refresh; secrets reveal only when you click the eye. (#352)
- **Microsoft Mail OAuth provider + MCP registry entry.** Read and send Outlook / Microsoft 365 mail through a new `microsoft-mail` provider in Link, wrapping `@softeria/ms-365-mcp-server --preset mail`. PKCE-only public client (no app secret), mail-only scopes (`User.Read`, `Mail.ReadWrite`, `Mail.Send`, `offline_access`). Personal MSA, work, and school accounts all sign in through `/common`. Calendar / Files / Teams will ship as separate providers to keep consent screens minimal. (#389)

### Changed

- **Chat surface polish.** Message actions (timestamp, menu) hug the bubble's right edge instead of stranding past wide tool cards. Inline timestamps drop seconds — hover for full date+time, or hold Alt for the full inline format. Fullscreen chat (Ctrl+F) keeps its rounded panel framing instead of going edge-to-edge. (#352)
- **Design system pass across `@atlas/ui` and the playground.** New shared CSS reset and `utilities.css` (CUBE-style `.text-*` size scale, `.text-bright` / `.text-faded` color helpers, `.button` block with variant/size modifiers) loaded from both `agent-playground` and `chat-replay`. `PageLayout` gains a "shell mode" — a workspace-level sidebar as a direct child of `Root` switches the layout to a responsive 2-column grid via `:has()`. Hand-rolled `font-size` / `color` rules across ~50 playground components migrate to the utility classes. (#387)
- **MCP catalog sorts alphabetically.** Previous order grouped by static-vs-runtime source, which was invisible since the tree renders without group headers. (#389)
- **Installed Studio builds default `FRIDAY_LOG_LEVEL=warn` instead of `debug`.** `debug` was flooding the daemon log with per-token MCP stream events. Existing `.env` customisations are preserved on upgrade. Bare CLI / `deno task atlas` still falls through to `debug` for local development. (#392)
- **`deno.compile.json` drops its vestigial `tasks:` block.** The file is consumed only by `deno compile`, which doesn't read tasks. The block had silently drifted from `deno.json`; `deno.json` is now the single source of truth for tasks. Closes #317. (#393)

### Fixed

- **Launcher self-recovers from crashes.** The macOS LaunchAgent plist has shipped with `KeepAlive=<false/>` since v0.0.x, so launchd never restarted on panic, segfault, OOM, or signal-kill — a crashed launcher left a vanished tray icon and a dead playground port until the next login. Now writes `KeepAlive={Crashed:true, SuccessfulExit:false}`, which catches any abnormal exit while leaving clean Quit / installer-SIGTERM alone. **macOS 26 (Tahoe) gotcha:** `{Crashed:true}` *alone* is silently ignored on Tahoe — verified empirically on 26.4.1 — so the two-key form is load-bearing on current macOS. Existing users on either the old `<false/>` form or the intermediate `{Crashed:true}` shape get migrated to the canonical two-key dict on their next launcher boot. (#345)
- **Two-pass MCP session eviction unsticks long-lived SSE streams.** The 60s cleanup tick filtered eviction candidates by "no active requests," but long-lived SSE / chat-storage streams keep that counter above zero for the lifetime of an open browser tab. Once the session map filled, the warning fired every 60s with no progress and new sessions got crowded out — the studio chat-storage "Offline" wedge required a daemon restart to recover. Now: prefer evicting idle sessions oldest-first; if still over the limit, force-evict the LRU active session and tear down its stream. Force-eviction logs at `error` (with the evicted session ids) since it disconnects a live client. (#369)


## [0.1.9] - 2026-05-18

Wire any external service to a workspace via webhooks. The tunnel no longer ships hardcoded `bitbucket` / `jira` / `github` providers — instead it hands the raw bytes to your workspace agent, which parses the body and verifies the signature itself. A new `wiring-external-webhooks` skill walks through the setup end-to-end (#354). **OpenRouter** joins as an opt-in LLM provider — set `OPENROUTER_API_KEY` and pick any tool-capable model from their catalog (#348). Installed **Friday Studio** stops hitting dead ports: workspace imports from the discover catalog and bundled-agent runs both work against the real daemon now (#386), and imported workspaces that reference bundled `type: user` agents actually run them instead of failing with "agent not found" (#349). Read **Breaking changes** before upgrading — webhook URLs have moved and the signal-trigger SDK response shape narrows to a discriminated union.

### Breaking changes

- **Webhook-tunnel collapsed to a single `raw` provider — `github`, `bitbucket`, `jira` are all gone from the binary.** `tools/webhook-tunnel/provider/provider.go` is now ~25 lines exposing `Names() []string { return []string{"raw"} }` / `IsValid(name string) bool { return name == "raw" }`; a new `TestIsValidRejectsRemovedProviders` in `handler_test.go` explicitly asserts the three dropped names are rejected. Deleted alongside: the embedded `provider/mappings.yml` allowlist (-76), `provider/extract.go` (`extractByPath`, `splitArraySegment`, -63) + `extract_test.go` (-92), `provider/handler.go` (`Verify` / `Transform` / event-filter pipeline, -143), and the entire `tools/webhook-tunnel/passphrase/` package (`passphrase.go` -32, `wordlist.go` -46, `passphrase_test.go` -34). The `Handler` interface, `Get(name)`, `List()`, `Init()`, `EventMapping`, `ProviderConfig`, and the `gopkg.in/yaml.v3` dependency are out of `provider/provider.go` entirely. Workspaces previously registered on `/hook/bitbucket/<workspaceId>/<signalId>`, `/hook/jira/...`, or `/hook/github/...` no longer resolve — webhooks must move to `/hook/raw/<workspaceId>/<signalId>` and the workspace agent parses + (optionally) HMAC-verifies the body against the bytes Friday now hands over byte-for-byte. The `/status` JSON response loses its `secret` field; `pattern` is now `"/hook/raw/{workspaceId}/{signalId}"`. The new `wiring-external-webhooks` skill is the canonical setup guide. (#354)
- **Webhook-tunnel `WEBHOOK_SECRET` + `WEBHOOK_MAPPINGS_PATH` env overrides + `Config.WebhookSecret` struct field removed.** No replacement at the tunnel layer — signature material rides through the proxied request headers (the tunnel deliberately keeps `authorization` / `cookie` because Stripe / Slack / etc. carry signature bytes there) and verification happens inside the workspace agent. Per-workspace secret storage belongs in the workspace's own `.env` overlay (the supply side shipped in 0.1.8). The `passphrase.Generate()` fallback that previously synthesized a `WEBHOOK_SECRET` is gone with the package. (#354)
- **Tunnel forwarder body cap lowered from 25 MiB → 4 MiB.** `tools/webhook-tunnel/forwarder/forwarder.go` switches from a `Forward(...)` helper that read+marshalled bodies into a buffer to two `httputil.ReverseProxy` returners — `WebhookProxyHandler()` (rewrites `/hook/raw/...` → `/api/workspaces/.../signals/...`) and `ProxyHandler()` (rewrites `/platform/<provider>[/<suffix>]` → `/signals/<provider>[/<suffix>]`, preserving query strings for Meta WhatsApp's `hub.verify_token` / `hub.challenge` handshake). Bytes are base64-encoded into the JetStream envelope on publish; the 4 MiB ceiling keeps the post-base64 message under the JetStream `max_msg_size` headroom (`DEFAULT_MAX_MSG_SIZE` in `signal-stream.ts` raised to 8 MiB in this release). A `Content-Length > maxBodySize` pre-stream 413 short-circuit lands in `handleHook` so oversized uploads fail fast instead of partially streaming. Hop-by-hop header stripping (RFC 7230 §6.1) now comes for free via the stdlib reverse proxy. (#354)
- **`WEBHOOK_TUNNEL_URL` env override removed.** `getWebhookTunnelUrl()` in `apps/atlasd/src/services/communicator-wiring.ts` now reads `EXTERNAL_TUNNEL_URL` only — the prior literal `process.env.WEBHOOK_TUNNEL_URL` is replaced. On installed Friday the launcher (`tools/friday-launcher/cert_env.go`) remaps webhook-tunnel to `FRIDAY_PORT_WEBHOOK_TUNNEL` (e.g. `:19090`) and exports `EXTERNAL_TUNNEL_URL` — but never `WEBHOOK_TUNNEL_URL`, so the prior variable silently fell through to a dead `:9090`. New `"uses EXTERNAL_TUNNEL_URL when the launcher exports it (installed rig)"` test asserts the bare 9090 default is never hit. (#390)
- **`WorkspaceRuntime.triggerSignalWithSession` / `triggerSignalWithResult` signatures changed.** Five trailing positional optionals (`streamId`, `onStreamEvent`, `skipStates`, `abortSignal`, `parentSessionId`) collapsed into a single `opts: TriggerSignalOpts = {}` exported from `@atlas/workspace`. The bag picks up a new `webhookContext: { body?: string; headers?: Record<string, string> }` field threaded into `engine.signal({ type, data, body, headers })`. The matching daemon-layer wrapper `triggerWorkspaceSignal` in `apps/atlasd/src/atlas-daemon.ts` (+65) takes the same shape. Out-of-tree daemon embedders calling these methods must migrate from positional → options-bag; internal call sites are updated in the same PR. (#354)
- **`AtlasClient.triggerSignal` return type narrows to a discriminated union; old `SignalTriggerResponseSchema` in `@atlas/client/schemas` deleted.** Returns either `{ status: "completed", sessionId, output?, summary? }` (sync default) or `{ status: "accepted", correlationId }` (async / `?nowait=true` / webhook mode). Callers must narrow on `status` before reading `sessionId` / `correlationId`. The canonical schema moved to a new `packages/core/src/signal-trigger-response.ts` (+32) and is re-exported from `@atlas/core` (preferred) and `@atlas/client` (back-compat one-line import path). On response drift the client now throws `AtlasApiError(..., 502)` instead of returning a malformed envelope; the previous non-discriminated `{ message, status, workspaceId, signalId }` shape is gone. (#354)

### Added

- **Webhook fast path + `?nowait=true` async mode on `POST /api/workspaces/<id>/signals/<sig>`.** `apps/atlasd/routes/workspaces/index.ts` (+345) reads raw bytes via `c.req.arrayBuffer()`, JSON-parses defensively, and runs a tight envelope discriminator — the request is webhook mode iff the body is non-empty AND has at least one key AND every key is in `SIGNAL_ENVELOPE_KEYS` (derived from `signalBodySchema.shape` at module load with an assertion that the set is non-empty, so a zod-upgrade can't silently collapse the discriminator and route every envelope request to webhook mode). Mixed-shape bodies like `{"payload": {...}, "action": "opened"}` route to webhook mode so zValidator can't strip upstream fields. Hop-by-hop headers (RFC 7230 §6.1 + `host` / `content-length`) are stripped before publish; `authorization` / `cookie` are deliberately kept because Stripe / Slack / etc. carry signature material there. The body is base64-encoded into the JetStream envelope as `webhookBody` + `webhookHeaders` (new optional fields on `SignalEnvelopeSchema`); route returns `202 { status: "accepted", correlationId }`. The same handler honors `?nowait=true` for non-webhook callers: publishes to SIGNALS and returns 202 immediately without subscribing to the response stream. `Accept: text/event-stream` still wins over `?nowait` (an `index.test.ts` test locks the priority). (#354)
- **`envelopeToWebhookContext(envelope)` + raised `DEFAULT_MAX_MSG_SIZE`** in `apps/atlasd/src/signal-stream.ts` (+55, plus a new 360-line `signal-stream.test.ts`). New pure helper returns `{ body, headers } | undefined` — explicitly `undefined` rather than `{}` when neither field is set, so non-webhook signals don't pollute `ctx.input.raw["body"]` with `""`. `SignalEnvelopeSchema` + `PublishSignalOpts` gain optional `webhookBody` (base64) and `webhookHeaders` (lowercased `Record<string, string>`). `DEFAULT_MAX_MSG_SIZE` raised from 1 MiB → 8 MiB to match `FRIDAY_JETSTREAM_MAX_MSG_SIZE` in `packages/jetstream/src/config.ts` — the old 1 MiB applied only to test-constructed streams and would have let a base64 webhook body pass tests but 500 against a real daemon. Tests round-trip webhook fields through publish → consume → schema and unit-cover all four `envelopeToWebhookContext` shapes. (#354)
- **`SignalTriggerResponseSchema` shipped from `@atlas/core` as the canonical signal-trigger response shape.** New `packages/core/src/signal-trigger-response.ts` (+32) + a `packages/core/src/signal-trigger-response.ts` export from `packages/core/mod.ts`. Discriminated union over `status` — `"completed"` carries `sessionId` + optional `output` + optional `summary` (sync); `"accepted"` carries `correlationId` (async / `?nowait` / webhook). Producer (atlasd routes) and every consumer (mcp-server `signals/trigger.ts`, atlas-cli `signals/trigger.ts`, workspace-chat `job-tools.ts`, `run-job-dialog.svelte`, `@atlas/client` SDK) zod-parse against the same source. (#354)
- **FSM engine carries webhook `body` / `headers` through `inputFrom` action chaining.** `Signal` and `Context.input` in `packages/fsm-engine/types.ts` (+31) gain optional `body?: string` / `headers?: Record<string, string>` siblings of `data` / `config`; `PrepareResultSchema` adds the same. `parsePrepareResult` no longer returns `undefined` when only `body`/`headers` are set (previously required `task` or `config`, silently dropping `__lastPrepare` carryover for webhook signals whose downstream code action returned no structural data). Two seams fixed in `fsm-engine.ts` (+44): initial `prepareResult` from a signal propagates `sig.body` / `sig.headers` to the top level (not nested under `config`); the `inputFrom` merge at the second-action-in-a-job path explicitly preserves `body` / `headers` from upstream — previously the `inputSnapshot` spread overwrote them with `undefined`, so a two-state `validate → process` job saw the body in action #1 but not action #2. New ~100-line case in `tests/prepare-result.test.ts` drives a two-state FSM and asserts both actions see the body. (#354)
- **OpenRouter as a sixth opt-in LLM provider.** New `packages/llm/src/openrouter.ts` (+38) wraps `@ai-sdk/openai` against `https://openrouter.ai/api/v1` via `createChatCompletionsProvider` (same rationale as LiteLLM — the Responses API breaks tool-use for non-OpenAI models). Set `OPENROUTER_API_KEY` and write `conversational: openrouter:anthropic/claude-sonnet-4.6` (or any tool-capable slug) in `friday.yml`. `OPENROUTER_PROXY_URL` is honored via `createProxyFetch`. The catalog hits `https://openrouter.ai/api/v1/models?supported_parameters=tools` so the picker isn't drowned by OpenRouter's 500+ slugs, and sends `HTTP-Referer: https://hellofriday.ai/` + `X-OpenRouter-Title: Friday` attribution headers so usage shows up on `openrouter.ai/rankings`. `REGISTRY_PROVIDERS` / `RegistryModelId` extended; `PROVIDER_ENV_VARS.openrouter = "OPENROUTER_API_KEY"`; `PROVIDER_META.openrouter` carries letter `R`, prefix `sk-or-v1-`, help URL `openrouter.ai/keys`. Default platform models untouched — pure opt-in. (#348)
- **`@friday/wiring-external-webhooks` system skill.** New 158-line `packages/system/skills/wiring-external-webhooks/SKILL.md` covering: URL shape (`https://<public-tunnel-url>/hook/raw/<workspaceId>/<signalId>` — explicit warning not to give upstreams the direct `/api/workspaces/.../signals/...` route, which isn't exposed via cloudflared); discovery via `curl -sk "${EXTERNAL_TUNNEL_URL:-https://localhost:9090}/status" | jq` (empty body → wrong port for this rig; `ps auxww | grep cloudflared` to read the actual `--url`); the `ctx.input.config or ctx.input.raw` landing site; the silent-strip foot-gun where a nested `type: object` without `additionalProperties: true` turns the body into `{}` for the agent; the tunnel error catalog (`400 Unknown provider`, `400 raw provider expects JSON object`, `400 empty body`, `413 body exceeds N bytes`, `502 Cannot reach atlasd`); and a `concurrency: skip` first-action guard. Cross-referenced from `writing-workspace-signals` (+11) and exercised end-to-end by a new `tools/qa/live-daemon/scenarios/webhook-setup-bitbucket.ts` (+1867) plus a matching promptfoo trio (`webhook-setup-bitbucket-{config,provider,tests}` +44/+103/+67) covering 8 contract checks. (#354)
- **Tier 0 "Reading Signal Payloads" section in `writing-friday-python-agents/references/examples.md`** (+81). Anti-pattern: `parse_input(prompt)` scrapes JSON from the enriched prompt string, which holds the action config (`{config: {prompt, outputTo, ...}}`), not the webhook body — wrong dict for webhook-direct invocations. Correct pattern: `payload = ctx.input.config or ctx.input.raw`. Includes a full wired `workspace.yml` example pinning `initial: idle` (not `initial: process`) — the "Missing sessionId in signal context" error users hit was caused by parking the agent in the initial-state entry actions, because signal context isn't bound to initial-state entries yet. (#354)

### Changed

- **Link GitHub provider accepts five new token prefixes.** `GitHubSecretSchema.access_token` `.refine()` in `apps/link/src/providers/github.ts` expanded from `ghp_` / `github_pat_` to also accept `gho_` (user-to-server), `ghu_` (user-to-user), `ghs_` (server-to-server), `ghr_` (refresh). Error message rewritten to enumerate all six prefixes. Enables saving tokens minted by `gh auth` / GitHub Apps, not just classic PATs. (#354)

### Breaking changes

- **`scratchpad` storage and tools removed.** Agents that need chunked working state should declare a `memory.own` store typed `short_term` and call `memory_save`/`memory_read` instead. ## [0.1.1] - 2026-05-01

### Added

- Workspace-scoped skill assignment tools (`assign_workspace_skill` / `unassign_workspace_skill`) for the workspace-chat agent. Calls `SkillStorage` directly, validates `@namespace/name` refs via `SkillRefSchema`, and is idempotent. Assignments take effect on the next chat turn because `resolveVisibleSkills` reads the assignment table every turn. System prompt and `workspace-api` skill reference updated to distinguish ephemeral `load_skill` from persistent workspace assignment. (#114)

### Fixed

- Hung MCP servers no longer block workspace chats. `Promise.allSettled` replaces the serial connect loop so one stuck server cannot stall the others, and hard timeouts are enforced at every phase: 4s on the HTTP reachability probe, 20s on `createMCPClient` + `listTools` handshake, and a 15-minute ceiling on `callTool` invocation. Timed-out servers are silently dropped; structured `operation: "mcp_timeout"` telemetry (with `serverId`, `phase`, `durationMs`) is logged for future circuit-breaker decisions. The previous retry on hang was removed — one shot, one timeout, move on. (#116)
- Four macOS launcher / tray bugs (#113):
  - Autostart plist now points at the bundle ID the installer actually writes (`ai.hellofriday.studio`), not the never-installed `-launcher` suffix. LaunchServices no longer fails autostart and the tray "Restart" action with `LSCopyApplicationURLsForBundleIdentifier`. Existing broken installs self-heal on next launcher start.
  - Tray "Restart all" no longer paints the menubar `Error` for the few seconds children take to come back up. A new restart-grace window (active during the in-flight restart and for `bucketFailGraceWindow` after) suppresses the transient `AnyFailed` bucket.
  - Tray "Restart all" no longer terminates the supervisor watchdog. `RestartAll` now calls `processRunner.RestartProcess` per service in `startOrder`, which keeps `pendingRestarts > 0` and prevents process-compose's `runner.Run` loop from exiting "Project completed" between the stop and start passes. As a side benefit, four of five services stay running at any instant during a user-initiated restart.
  - The user's "Start at login" disabled choice now survives launcher restarts. `isAutostartStale()` on darwin matches the Windows contract: stale iff the registered value is both non-empty and mismatched. An absent plist means user-disabled and is left alone.
  - The cold-start grace and post-restart grace are unified behind a single `bucketFailGraceWindow` constant (90s) that covers the readiness-probe budget (`InitialDelay=2s + FailureThreshold=30 × PeriodSeconds=2s = 62s`), so legitimately slow first-pass readiness (the friday daemon's ~24s workspace scan + skill bundle hashing + cron registration on cold start) no longer flips the bucket red mid-startup.
