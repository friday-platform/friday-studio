## [0.1.4] - 2026-05-08

### Breaking changes

- **`scratchpad` storage and tools removed.** The schema-unification effort that gated MCP-exposed scratchpad has been resolved by deletion: `packages/agent-sdk/src/scratchpad-adapter.ts`, `packages/agent-sdk/src/platform-tools/scratchpad-tools.ts`, the entire `packages/adapters-memory/` package, the `/api/scratchpad` HTTP route, `scratchpad.ts` storage module, `ScratchpadWriteEventSchema`, and the `scratchpad_append/read/clear` entries from `PLATFORM_TOOL_NAMES` are all gone. Agents that need chunked working state should declare a `memory.own` store typed `short_term` and call `memory_save`/`memory_read` instead. `atlas-cli inspect --kind scratchpad` arm dropped. Existing daemons get the orphaned `KV_SCRATCHPAD` bucket removed by a new idempotent migration `m_20260507_120000_drop_scratchpad_kv`. (#225)
- **`outputTo` LLM actions must finish via the `complete` tool.** Free-form `outputTo` actions now inject a mechanical `complete` tool contract even when `outputType:` is absent; no-tool actions pin `toolChoice` directly to `complete`, tool-using actions run with `toolChoice: required` and a hard post-call `complete` requirement. Authors who previously relied on the LLM ending in plain prose for `outputTo` will see actions fail validation until they emit through `complete`. Closes a class of empty-`response` regressions where `record_validation` or another final tool would persist an empty stub. (#225)
- **`validate.threshold` and `successSignal` schema fields removed.** `ValidateStrategySchema`'s object-form `threshold` enum was parsed but never read; `SessionSummarySchema.successSignal` plus `SuccessSignalSchema` were reserved-but-never-populated. Both gone. Workspaces that hand-coded either field will fail validation. (#225)
- **`OutputValidator` / `validateOutput` engine option removed.** `FSMEngineOptions.validateOutput`, the `legacyValidatorShim` adapter, and the `OutputValidator` / `LLMOutputValidationResult` types are deleted; `validate: external` now exclusively routes through the delegate-driven `@friday/judge-agent`. The `packages/hallucination` package shrinks by ~3,170 LOC (loses `detector.ts`, `fsm-validator.ts`, and integration tests; keeps a narrower `verdict.ts` + `supervision-levels.ts`). The `retryOnFail` field is dropped from the `validate:` schema — wrap actions in an FSM-level retry pattern instead. (#225)
- **`LocalSessionHistoryAdapter` deleted.** Filesystem-JSONL session history adapter gone (~470 LOC including its dedicated test); production has wired `JetStreamSessionHistoryAdapter` exclusively since 0.1.2. The export from `packages/core/mod.ts` is gone; consumers should switch to the JetStream adapter or an inline in-memory mock for tests. (#225)

### Migration

- **`ELICITATIONS` JetStream stream now requires `allow_msg_ttl: true`.** Daemons that ran an earlier broken migration created the stream without this flag, so on NATS 2.11+ every `request_tool_access` and `request_human_input` publish failed silently with "per-message ttl is disabled". A pre-flight `bootstrapElicitationsStream(nc)` runs at daemon boot ahead of HTTP route mount: missing/misconfigured streams self-heal in-place via `streams.update`; daemons that already ran the broken migration fix themselves on next startup. The adapter no longer self-creates the stream — it surfaces a clear "re-run migration" error if the bootstrap didn't run. (#225)
- **`SESSION_EVENTS` JetStream `duplicate_window` raised from 2m default to 24h.** Long-running FSM jobs (auto-triage > 2 min, review-inbox 63s, chat-flip ~150s) outran the 2-minute dedup window: re-published stable-msg-id step events landed as new entries, the reducer appended duplicate `AgentBlock`s, and status derivation surfaced sessions as `active` post-completion. Existing streams reconcile automatically via `streams.update`; reducer also got a defense-in-depth no-op for `(stepNumber, agentName)` duplicates from any pre-fix events still in stream history. (#225)

### Added

- **Durable Human-in-the-Loop (HITL) primitives.** Two new platform MCP tools — `request_tool_access(toolName, reason)` and `request_human_input(question, options?, multi_select?)` — turn LLM-blocking events into durable elicitations the user answers from the Activity feed. Both tools subscribe to NATS `elicitations.<wsId>.<sid>.<id>` for terminal-state notifications, fall through to KV polling on missed publishes, and return `granted/declined/expired/answer` semantics to the blocked action. Backed by a new `packages/core/src/elicitations/` module (Zod schema + JetStream adapter; `ELICITATIONS` stream + `ELICITATION_STATUS` KV), HTTP routes at `/api/elicitations` (`GET /`, `GET /:id`, `GET /stream` workspace-scoped SSE, `POST /:id/{answer,decline}`), a 60s pending→expired sweeper with read-time derivation, and CAS-fenced terminal transitions so concurrent answers race-safely. Allow-always tool-access grants persist in workspace-scoped KV and short-circuit before creating new elicitations. `expiresAt` is derived from job-scoped `config.timeout` when set, falling back to `DEFAULT_ELICITATION_TTL_MS` (30 min). (#225)
- **Activity page in agent-playground.** New SvelteKit routes at `/activity` (global) and `/workspaces/:id/activity` (scoped) list pending + recent elicitations from `GET /api/elicitations`; live updates over a workspace-scoped SSE stream on the scoped page, 5s polling on the global page (deliberately avoids opening one EventSource per workspace, which exhausted the per-origin browser request pool). Filters by status / workspace / kind; expired elicitations render read-only. Sidebar pending-count badges share the Activity TanStack Query cache and update without reload. Includes nested-choice rendering for repeated `request_human_input` prompts (parses `[n]` labels and `n:action` values into structured per-item rows; selection emitted as a JSON-array answer string for backend compatibility). (#225)
- **FSM action outputs persisted as artifact references mid-session.** `outputTo` documents are written through `ArtifactStorage` immediately after each action completes (not just at session-complete), so downstream `inputFrom` steps consume compact `{artifactId, summary}` refs and the next LLM call hydrates content via `expandArtifactRefsInInput`. Adds `session.artifactRefs` (additive; existing `session.artifacts` preserved); titles/summaries use the action-declared `summary:` field when present, otherwise synthesized from a 300-char `JSON.stringify(doc.data)` truncation. Persistence is best-effort: artifact-create errors warn and skip rather than failing the session; the document store remains the source of truth. (#225)
- **Job-tool result shape carries `artifactIds + summary` (the supervisor flip).** SSE `job-complete` and the workspace-chat job-tool now return `{success, sessionId, status, artifactIds, summary}` instead of the full `Document[]`; the chat-supervisor's next-turn input drops from full job output to refs + summary, with `parse_artifact(<id>)` available for selective fetch. 3-tier summary fallback: `aiSummary.summary` → terminal-state `action.summary` → truncated `outputTo` data. Legacy `output: Document[]` still emitted alongside for back-compat with non-migrated consumers. The 50-email auto-triage scenario the plan was built around now returns ~"Processed 50 emails, 45 actions, 5 flagged" + 50 artifact IDs to chat instead of inflating the supervisor context. (#225)
- **Per-action skill allowlist (`skills:` field).** `LLMAction.skills` and `AgentAction.skills` were `@experimental` and silently ignored; now wired as a runtime narrowing filter on top of `resolveVisibleSkills` — `undefined` inherits, `[]` opts out (no `load_skill` tool), `["a","b"]` whitelists within the resolved set. Lives in a testable helper (`applySkillAllowlist` + `unmatchedAllowlistEntries`). `@experimental` note dropped from both schema docstrings. (#225)
- **Per-action `tools:` allowlist + auto-injected platform tools.** FSM `type: llm` actions auto-inject `memory_save`/`memory_read`, `scratchpad_*`, `artifacts_*`, `fs_*`, `bash`, `csv`, `webfetch`, `state_*`, `request_tool_access`, `request_human_input`, `load_skill`, and `delegate` (platform built-ins are ambient — `action.tools` narrows MCP-server tools only). Authors no longer redeclare these. `tools: []` locks down to the pinned set; auto-injected tools sit additively on top of authored ones. (#225)
- **`delegate` available as opt-in tool for FSM `type: llm` actions.** When an action declares `tools: [..., "delegate"]`, the engine constructs a sub-agent via `createDelegateTool` with a synthetic `UIMessageStreamWriter` that bridges chunks into the FSM signal's `onStreamEvent`. Honors workspace `delegation.max_depth` (default cap 1); skipped silently with debug log if `platformModels` isn't wired or the depth cap is met. `delegate/` moved from `workspace-chat/tools/` to `@atlas/core/delegate` so chat and FSM share a single import without layering violations. Grandchildren build a fresh delegate via `createDelegateTool({...deps, depth: depth+1})` per call rather than inheriting the parent's closure-bound depth. (#225)
- **Delegation budget enforcement (`delegation:` workspace + per-job block).** New `delegation: { max_depth, max_steps_per_call, max_output_tokens, max_input_tokens, max_wall_time_ms, max_cost_usd }` block at workspace + per-job levels. Promotes the hardcoded `CHILD_STEP_BUDGET` / `CHILD_MAX_OUTPUT_TOKENS` constants to config-driven and adds wall-clock (`AbortSignal.timeout`), input-token (per-step usage tally), and depth (parent-context counter) enforcement. On exhaustion, delegate returns `{ ok: false, reason: "budget_exhausted: <which>" }`. `max_cost_usd` is reserved (not enforced). Defaults: `max_depth: 1`, `max_steps: 40`, `max_output_tokens: 20000`, others `Infinity`. (#225)
- **Permissions block (`permissions: { dangerouslySkipAllowlist }`) at workspace + per-job level.** Mirrors Claude Code's `--dangerously-skip-permissions`. Three-level precedence merged via `resolvePermissions`: job > workspace > daemon `FRIDAY_DANGEROUSLY_SKIP_PERMISSIONS` env var. Each level can override its parent in either direction (a strict job inside a permissive workspace, or vice versa). When bypass resolves true, FSM `buildTools` skips per-agent tool narrowing and `request_tool_access` short-circuits with `{granted: true, reason: "bypass"}`. Schema lives in `packages/config/src/permissions.ts` (extracted to avoid the workspace.ts ↔ jobs.ts circular import). (#225)
- **Validation strategy contract (`validate:` action field + workspace/job defaults).** Authors pick `skip | self | external | auto` (or an object form `{ strategy, skill, agent }`) per action; defaults to `auto`, which routes through a new pure-function classifier (`READ_ONLY_ALLOWLIST` + `MUTATING_VERB_RE`) that picks `skip` for read-only/structured emits and `self` for mutating-tool/prose-emitting actions. Workspace and per-job `validation: { default, skill }` blocks merge with action-level via `resolveValidation` (precedence: action > job > workspace > `auto`). `external` fires only on explicit opt-in — most workloads pay zero validation cost vs. the previous default-judge pipeline. Per-MCP-server `validation: "read-only" | "mutating"` overrides on `WorkspaceMCPServerConfig` honor author intent over the regex defaults; `LLMAction.run_code: { readOnly: bool }` lets a deterministic `SELECT`/`GET` resolve to `skip` when paired with `outputType:`. (#225)
- **`@friday/validating-llm-outputs` system skill + inline self-check.** Powers `validate: self`: the runtime composes the skill body into the action's system prompt + auto-injects a new `record_validation` platform tool when decision === `self`. Verdict `blocking` triggers `failStep` semantics. Skill marked `user-invocable: false` so it doesn't appear in `load_skill` autocomplete. `validate: { skill: "..." }` per-action override lets authors point at domain-specific validators (financial-claims, medical-claims, etc.). Skill prompt orders output FIRST then `record_validation` to prevent the "Now let me record validation..." termination pattern; runtime sentinel-text guard at the outputDoc write site warns when persisted `response` looks transitional. `composeValidationBlock` lives in `packages/core/src/agent-context/` alongside `composeMemoryBlocks` and `composeArtifactBlocks`; carries a circuit breaker that escalates to error after 5 consecutive failures and resets on first success. (#225)
- **External validation via `@friday/judge-agent` delegate.** Replaces the runtime `validateOutput` hook with a sub-agent invocation; judge sees action output + a tool-call manifest with `resultArtifactId` for scrubber-lifted results (refs-not-bytes handoff — cost scales with judgment work, not input size). Phase 8 budgets apply automatically. `artifacts_get` + `parse_artifact` are scoped into the judge for selective pull of lifted bytes. Authors swap judge agents per-action via `validate: { agent: "..." }` for domain-specific validation; structured-output verdict shape pinned via `ValidationVerdictSchema` at the synthetic finish tool. (#225)
- **`step:complete.validation` first-class field.** Every `type: llm` action's `step:complete` now carries `{ strategy: "skip", skipReason }` / `{ strategy: "self", verdict, issues? }` / `{ strategy: "external", verdict, issues? }`. Structured + self emits implicit `{strategy: "self", verdict: "pass"}` on successful complete-tool emission. Existing `step:validation` events keep firing on external for back-compat. Authors can grep daemon logs for "how many blocking verdicts" / "what skipped and why"; `friday-cli` docs gain matching recipes. (#225)
- **`FSMEngine.initialize()` emits one info log per `(state × llm/agent action)`** summarizing the statically-resolved validate decision (`decision`, `source`, `reason`, `skill`). Authors can grep daemon logs to see what's resolving where without bisecting the workspace > job > action precedence chain. (#225)
- **Ephemeral lifecycle for FSM artifacts and notes (`lifecycle: { kind, boundTo? }`).** Artifacts and narrative-memory entries gain a lifecycle kind (`durable` | `ephemeral`) with optional `boundTo` for session binding. Default policy: FSM action outputs from non-terminal states + `notes` (short_term memory) → ephemeral, bound to session; outputs from terminal states + `memory` (long_term) → durable. Workspace.yml overrides: `jobs.<name>.artifacts.ephemeral: bool`, `memory.own[].ttl: Duration`. (#225)
- **Ephemeral artifact grace-window sweeper.** New `apps/atlasd/src/sweepers/artifacts-sweeper.ts` runs hourly (configurable via `FRIDAY_SWEEPER_INTERVAL_MS`). At session-complete each ephemeral artifact gets `expiresAt = completedAt + grace`; default 24h, overridable via `artifacts: { default_grace }` at workspace + per-job level. The sweeper scans expired artifacts for inbound references — memory-entry text containing the artifact id, `aiSummary.keyDetails` URL pointing at it — and promotes to durable via lifecycle update; otherwise deletes. Reference-scan helper at `packages/core/src/artifacts/reference-scan.ts`. (#225)
- **Retrieval-gated artifact injection in FSM/chat prompts.** Both chat and FSM `type: llm` actions now inject a `<retrieved_content provenance="..." origin="..." fetched_at="...">` block carrying recent session artifacts (up to 10 per session, queried via the new `ArtifactStorageAdapter.listBySession` at action-start). Each block carries summary + artifactId so the LLM can `parse_artifact` for full content. `composeArtifactBlocks` lives alongside `composeMemoryBlocks` in `packages/core/src/agent-context/compose-blocks.ts`. (#225)
- **Scrubber lifts large text/JSON tool results to artifacts.** Closes the original Phase 3 intent: text/JSON tool results above 8 KB now lift to artifacts before reaching the LLM message buffer (binary lifting was already shipped). Mime-sniffed via prefix patterns (`text/html`, `application/json`, `text/markdown`, `text/plain`). Lifting moved off the MCP-tool-result boundary onto the persistence layer (`liftToolResultsForPersist` helper) — producer LLM operates on inline bytes (no bail-prose pattern), persistence and cross-consumer handoff stay compact. Marker text (`[attachment lifted to artifact <id>]`) shape unchanged so downstream consumers (G2 marker rendering, session events, agentBlocks, cross-action `inputFrom`) keep working. Agent-playground tool-call cards auto-render lifted-artifact markers via the existing artifact preview component (multiple markers render as a list; fetch failure falls back to raw marker text). (#225)
- **Async `aiSummary` off the critical path + fast-path from terminal `action.summary`.** When the terminal action declares `summary:` AND has structured `outputType:`, runtime builds `SessionAISummary` synchronously from the declared summary + leaf-field-derived `keyDetails` and skips `generateSessionSummary` entirely (saves ~1–2s for any author who declared `summary:`). Otherwise `generateSessionSummary` runs as a detached promise after `sessionStream.finalize()` — SSE `job-complete` responds immediately with the action.summary fallback (or truncated `outputTo` data); the LLM-generated `aiSummary` writes to session metadata asynchronously and emits a `session:summary` event for live subscribers. Saves 1–2s per job, every job. (#225)
- **Session provenance + token usage on every step.** `SessionSummary` gains optional `parentSessionId` / `parentEventId` populated from chat-spawned-job, delegate-spawned-child, and signal-trigger paths so the daemon can walk a chat→job→child tree from the root. `step:complete` events now persist `usage: { inputTokens, outputTokens, cacheReadTokens, cacheWriteTokens, model }`. `AgentBlock.usage` aggregates from `step:complete.usage` so the `workspace-chat` agent's token usage stops being silently dropped in nested-agent flows. (#225)
- **Python user-agents receive structured FSM input via `ctx.input`.** FSM action input (including `inputFrom` artifact-ref hydration for compact handoff) is threaded into the Python subprocess agent execution path; the bundled `friday-agent-sdk` Python pin moved to `0.1.7` across launcher (`tools/friday-launcher/paths.go`), Docker, and Studio installer prewarm paths. Active `FRIDAY_NATS_URL` is forwarded into subprocess agent execution + registration so harness daemons on random NATS ports can run user agents (was hardcoded to `localhost:4222`). User-agent `agent.py` paths can now call `ctx.tools.list()` / `ctx.tools.call()` against host-injected platform tools (including `request_human_input`); `AgentSessionDataSchema` gained the new scope fields in both `@atlas/agent-sdk` and `AgentServer`'s local copy. (#225)
- **`writing-friday-python-agents` + `repairing-workspaces-agents` system skills.** New runbook skill encodes the production-derived repair sequence for upgrading existing workspaces, jobs, user agents, and generated agents to current runtime contracts (inspect evidence, load companion skills, identify owning boundary, replace direct MCP/daemon calls with host capabilities, restore `outputTo`/`complete` contracts, use `request_human_input` for HITL). (#225)
- **Live-daemon QA harness + first-principles eval suite.** New `tools/qa/live-daemon/harness.ts` spawns a fresh `atlasd` against an isolated `FRIDAY_HOME` + isolated NATS server (no longer reuses ambient `localhost:4222`), registers fixture workspaces, fires signals, parses SSE, and walks raw session events for token-usage aggregation. Twenty-plus first-principles scenarios cover scrubber lift threshold, platform-tool parity, ephemeral artifact lifecycle, delegate isolation, delegate budget exhaustion, retrieval injection, request_tool_access bypass + denial + unknown-tool, request_human_input answered/declined/expired, supervisor flip compactness, chat follow-up compactness, Python user-agent inputFrom + tool calls, bundled `type: atlas` agent output, Activity sidebar UI, structured `validate: self` verdict shape, and many more. Promptfoo wrapper (`tools/qa/live-daemon/promptfoo/`) memoizes harness runs across concurrent providers. Two new `workspace-chat` evals (`prompt-behavior.eval.ts` + `elicitation-behavior.eval.ts`) gated on `ANTHROPIC_API_KEY`. The obsolete phase-numbered `all-phases` / `phases-3-9` / `auto-triage-baseline` / `chat-flip` runner + `inbox-corpus` fixtures + committed result artifacts are removed; first-principles is the new source-of-record. (#225)
- **Sanitized global elicitation SSE feed.** New `GET /api/elicitations/stream` (in `apps/atlasd/routes/elicitations/index.ts`) emits a sanitized summary schema for `elicitations.>` (defined in `packages/core/src/elicitations/model.ts`) — only safe metadata, no full envelopes. The app root mounts a single `elicitation-global-stream.svelte` subscriber that refetches affected list caches; this replaces both the old global 5s polling and the per-workspace EventSource fanout that #229 had to disable. Workspace chat/Activity streams keep using workspace-scoped full envelopes for inline HITL forms. (#230)
- **Deterministic `create-note` and `noop` ops in the bundled hubspot agent.** `HubSpotOperationSchema` gains two variants the dispatcher recognises before falling through to the LLM. `create-note` posts a CRM Note (`hs_note_body` + `hs_timestamp`) attached to a ticket via the existing `batchCreateCrmObjects` helper, reusing `DEFAULT_ASSOCIATION_TYPES.notes.tickets = 228` (the standard notes-tickets association id) and propagating `numErrors` / `errors[]` / `noteId` into structured `data`. Body is passed through byte-verbatim; `hsTimestamp` defaults to `new Date().toISOString()` when missing or empty. `noop` is a success-with-no-side-effects skip sentinel that returns immediately without spawning an SDK Client — lets upstream picker agents emit a "do nothing" envelope on the empty-cron-tick path without a separate FSM state. Schema rejects empty-string `ticketId` / `body` (`.min(1)`) so malformed envelopes fall through to the LLM instead of being forwarded for late SDK rejection. Eliminates the last LLM round-trip in the bucketlist-cs auto-answer drain pipeline. (#220)
- **Deterministic `read-ticket` op in the bundled hubspot agent.** New variant on `HubSpotOperationSchema` + dispatcher case in `packages/bundled-agents/src/hubspot/agent.ts` that fetches the ticket via the SDK directly and returns structured `data.{ticketId, subject, body, priority, pipelineStage, pipeline, createdAt, companyId, companyName, contactIds, threadId, transcript, skipped}`. The `response` field stays a one-line human status (e.g. `"Read ticket 310814009050 (NFL)."`) — explicitly NOT structured data in markdown form, which would re-create the regex-scraping fragility the op was built to eliminate. Empty `ticketId` (the `{done: true}` skip sentinel from upstream picker agents) emits a skip envelope without hitting HubSpot. `includeTranscript` is reserved on the schema for a future second-hop fetch of conversation threads + ticket-associated emails. Replaces a 40-line LLM tool-call prompt with a one-line envelope, drops cron runtime by ~30s/tick, and lets five downstream user agents (synthesizer, bucketlist-config-loader, briefing-builder, reply-builder, draft-formatter) read structured fields from `data.*` instead of regex-scraping prose. (#228)

### Changed

- **Validation injection happens at the LLM-prompt-assembly site, not at the FSM handler** — closes the `case "llm"` vs `case "agent" → type: llm` asymmetry. `convertLLMToAgent` in `packages/core/src/agent-conversion/from-llm.ts` calls `composeValidationBlock` identically to the inline FSM path. Authors who refactored inline actions into reusable workspace agents no longer accidentally turn off validation. `__atlas_validate` reserved key on `AgentContext.config` carries the resolved decision/skill across the runtime/MCP boundary; `type: user` and `type: atlas` resolution paths skip naturally (Python is code; bundled agents have fixed prompts). (#225)
- **Cascade-result derivation removed from completion caches.** `runtime.completedSessionMetadata` + `completedSessionDocuments` are gone as cascade-result sources; `getSessionJobResult` now returns directly from the live `SessionResult` before `finalizeSession` tears down per-signal state. The artifact sweeper reads `aiSummary` references from durable session history (not live runtime memory). `midSessionArtifactRefs` retained only as an active-session projection, cleared on failure/shutdown. (#225)
- **`FSMEngine` action outputs are document-canonical.** Documents are now the single output surface; `context.results` / `engine.results` derive as a projection plus auxiliary seeded values rather than dual-writing. LLM-backed agent actions with `outputTo` get a mechanical `complete`-tool output contract (default non-empty `response` schema when no explicit `outputType` declared) so validation/tool calls can no longer persist an empty stub. `convertLLMToAgent` pins runtime `outputSchema` to the `complete` tool. (#225)
- **`inputFrom` owns its artifact refs explicitly.** Removed the implicit post-action `artifactRefs` carry-forward shim — single and array `inputFrom` inputs now hydrate from the documents they actually reference instead of inheriting stale refs from prior `prepare` state. Closes a class of "second input artifact unavailable" bugs where the model treated artifact ids as raw data. (#225)
- **Workspace-chat job-tool bypasses cascade concurrency for chat-spawned jobs.** The `correlated` signal route was inheriting per-signal duplicate skipping + the 600s `awaitSignalCompletion` timeout for interactive chat flows that already own the parent chat stream. A per-daemon internal token (`X-Friday-Internal: <token>`) marks chat-spawned signal invocations as internal interactive requests and runs them directly; public HTTP signal triggers stay on the configured concurrency policy. Prevents one blocked HITL trip-planning job from rejecting another chat job. (#225)
- **Agent execution emits canonical projections directly on FSM action events.** Removed `WorkspaceRuntime.agentResultSideChannel` map; FSM action events carry the canonical agent-result projection and workspace runtime maps it to `step:complete` from event data. Tool-result lifting moved into the FSM agent path to match the inline LLM path. (#225)
- **Workspace-chat ambient artifact catalog injection removed.** Top-N artifact list injection at chat-system-prompt time is gone — explicit `artifactIds` and same-chat retrieval-gated content (the `<retrieved_content>` block) are now the only sources of artifact context. Eliminates a class of stale/irrelevant injected artifacts polluting the supervisor's context. (#225)
- **`completedSessionMetadata` now consumed-on-read (cap 1,000, was 10,000).** The cascade is the sole consumer of the metadata map; `getSessionJobResult` deletes both `completedSessionMetadata` + `completedSessionDocuments` after the read, making consumption-time GC the primary mechanism and shrinking the band-aid cap from `10_000` back to `1_000` (warn-log preserved on eviction). (#225)
- **Workspace-chat `request_tool_access` registered as a chat-side primaryTool.** Phase 12.C originally registered `request_tool_access` only via the platform MCP — chat composes `primaryTools` from chat-side factories and didn't see it. New `packages/system/agents/workspace-chat/tools/request-tool-access.ts` mirrors the MCP tool's behavior (bypass branch + elicitation emit) wired into `workspace-chat.agent.ts` `primaryTools`. (#225)
- **`registerReconnectReset` helper in `@atlas/jetstream`** — process-wide registry of "ensure-state" reset callbacks. Subscribes once to each `NatsConnection`'s status iterator; on reconnect, fans out a reset to all registered adapters. Wired into 4 adapters (artifacts, elicitations, mcp-registry, session-history) so the first post-bounce access re-provisions instead of failing with stream-not-found. (#225)
- **`fsm-engine` README rewritten around the current action model.** Stale `type: code` action / guard function / inline tool definition documentation removed; replaced with current-shape examples (`llm`/`agent`/`notification`/`emit`). Dropped the fictional programmatic-builder section (`createFSM`/`state`/`transition`/`codeAction` were never exported). (#225)

- **Phantom bundled-agent references now hard-fail at validation.** Workspace YAMLs with `type: atlas, agent: "<id>"` referencing a non-existent bundled agent (e.g., the deleted `email` agent) previously survived load: the daemon logged "Base agent not found" and silently dropped the agent at registration, so any FSM step delegating to it stalled in production with no surfaced error. - **JetStream store relocates from `$TMPDIR` to `<friday_home>/nats` automatically.** macOS periodically garbage-collects `$TMPDIR`, silently wiping chat history, memory, and workspace state. ### Changed

- **Chat SSE resume survives Chrome's ~50–60s fetch streaming cap.** Server emits `id: <index>` per SSE frame and accepts `Last-Event-ID` headers on resume; the client wraps `DefaultChatTransport`'s fetch with a `TransformStream` scanner that commits cursors only on event terminators. ### Fixed

- **Chat handler buffer-capture race.** The handler captured `ownBuffer = streamRegistry.getStream(chatId)` *after* awaiting `thread.subscribe` and `ChatStorage.appendMessage` — in that 10–50ms window, a follow-up POST could replace the chatId-keyed buffer, causing the handler to write turn 1's late chunks into turn 2's buffer. Now stashes the buffer on `Message.raw.turnBuffer` at dispatch and reads it back deterministically; non-web adapters (Slack / Telegram / Teams) that never create a buffer no longer log spurious `stream_event_dropped` warnings. (#206, follow-up to #192/#193)
- **Cross-turn event leak in chat stream registry.** Two cooperating bugs let a stopped turn's late events corrupt the next turn's stream: `finishStream(chatId)` was called in `finally` without identity verification, so a fresh buffer for the same chatId was flipped to inactive by the prior turn's cleanup; `appendEvent` looked up buffers by chatId only, so late events from an aborted turn (subprocess emit, NATS reconnect-buffer flush, streamSub drain) landed in whatever buffer was current. Now the handler captures `ownBuffer` once and threads it through both `appendEvent(chatId, chunk, ownBuffer)` and `finishStreamIfCurrent(chatId, ownBuffer)` so the identity check lives at the registry boundary. (#194)
- **`{{inputs.x}}` interpolation in agent action prompts.** Friday workspaces (which exclusively use agent actions) previously saw literal `{{inputs.description}}` in the prompt while the resolved values sat in an appended `## Input` block. Placeholders are now interpolated uniformly across both action types. (#199)
- **`cancelSession` no longer wedges when an engine ignores `AbortSignal`.** Workspace agents that make blocking calls without piping the AbortSignal through (e.g., `fetch` with no `signal:`) used to pin cascade-stream's in-flight slot forever — every subsequent trigger of the same signal was rejected as `skipped-duplicate` until daemon restart. New `awaitWithAbort` helper races `engine.signal()` against the effective AbortSignal, releasing the slot within ~1 tick on cancel; the orphaned `engine.signal` continues in the background with a tail handler that logs settlement and prevents `UnhandledPromiseRejection`. A `finalized` flag short-circuits late events so they don't land in JetStream after `session:complete`. (#202)
- **MCP cold-start probe timeout on first install.** The 5-second foreground probe was too tight for `npx` / `uvx` MCP servers on first run — npm cold-downloaded the package in 5–6s, timing out the user's first click while the second succeeded against cached npm. The new prewarm + cache pipeline (see Added) eliminates the cold-click failure mode. (#195)
- **FSM-path MCP servers now receive registry `platformEnv` variables.** The code-execution path through `runtime.ts` was handing raw workspace MCP server configs to the FSM engine without applying registry-owned `platformEnv` (`MCP_ENABLE_OAUTH21`, dummy `GOOGLE_OAUTH_CLIENT_ID` / `SECRET`, `WORKSPACE_MCP_STATELESS_MODE`). Cron-triggered LLM actions spawned `workspace-mcp` without those vars, so it ran in native-OAuth mode and rejected every bearer-authed request — meanwhile chat (via `discoverMCPServers`, which does apply `platformEnv`) worked fine. Mirrors the same pattern used in `executeCodeAgent`. (#196)
- **MCP subprocess registry evicts stale children on spec drift.** `sharedMCPProcesses.acquire` was first-spawner-wins: whichever caller hit `acquire(serverId, …)` first locked their `spec.env` for the daemon's lifetime, so a cached subprocess spawned by a buggy code path served stale config until restart. Each cached entry now carries a stable hash of `{command, args, env}`; on mismatch the registry SIGTERMs the child, waits up to 2s, SIGKILLs as fallback, and awaits full exit before respawn (mandatory because `workspace-mcp` binds fixed ports — a racing respawn through `TIME_WAIT` is the exact failure this registry exists to avoid). (#196)
- **MCP registry search tolerates unknown `registryType`.** `/api/mcp-registry/search` was returning 502 for any query matching a server with a `registryType` outside the hardcoded Zod enum (e.g., `nuget`); upstream defines the field as an open string, so a single unrecognised value tanked the entire response. Each entry is now parsed independently — malformed ones are dropped with logging — and the response shape gains a `dropped` count for telemetry. (#191)
- **Bundled-agent lookup now uses `Set.has` instead of the `in` operator.** `in` walks the prototype chain, so `agent: "toString"` / `"constructor"` / `"hasOwnProperty"` silently passed validation — the same shape of silent-failure bug the four-layer phantom-ref defence (above) prevents. Switched to a `Set` built from `Object.keys()` for safe own-property lookup. (#201)
- **`.env` quote handling for API keys and secrets.** The Settings UI's `PUT /api/config/env` used `@std/dotenv`'s `stringify`, which wrapped any value containing non-word chars (e.g., `-` in `sk-ant-foo`) in single quotes. The launcher read those values raw and passed them to spawned services, so agents inherited `ANTHROPIC_API_KEY='sk-ant-foo'` with literal quotes, breaking authentication. Values are now serialised unquoted unless truly necessary (whitespace, `#`, `$`, leading quote, newline, backslash); the launcher strips one layer of surrounding matching quotes on read so existing hand-quoted or bug-quoted files continue working. Keys are validated as POSIX identifiers and CR/LF in values is rejected at the boundary to prevent env-var injection. (#203)
- **`.env` parsing now trims trailing `\r` on CRLF files.** Both Rust (`parse_env_lines`) and the Go launcher (`loadDotEnv`) strip trailing `\r` so Windows-edited `.env` files parse correctly; internal CR is preserved. (#214)
- **Platform env vars refresh on every install / update.** Previous `write_env_file` only ran when the API Keys step rendered, so `FRIDAY_JETSTREAM_STORE_DIR` and other platform defaults wouldn't propagate on quiet updates. New `ensure_platform_env_vars` Tauri command runs unconditionally in `Extract.svelte` with add-if-missing semantics — never overwrites, only appends new keys. (#214)
- **`atlas migrate` reads `.env` for `FRIDAY_PORT_FRIDAY` daemon-probe.** Was hardcoded to 8080, broken in installer mode where the daemon binds to a configurable port. Now reads `.env` via `@std/dotenv`. (#214)
- **`studio-release` shell parser absorbed trailing ellipsis.** Under some locales bash treats UTF-8 continuation bytes as identifier characters, so `$publish_wf…` was parsed as including the ellipsis bytes, tripped `set -u`, and failed the publish workflow. Bracing as `${publish_wf}…` stops the parser at the `}`. (#190)

### Configuration

- **`FRIDAY_JETSTREAM_STORE_DIR`** (new) — absolute path to the JetStream store directory. Installer writes this on every install / reinstall; daemon and launcher read it for nats-server's `-sd` argument; falls back to `<friday_home>/nats` for operators on legacy `.env` files. (#214)
