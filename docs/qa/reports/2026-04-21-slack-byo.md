## Findings

### Finding 1 (P1, fixed in this run): atlasd `/signals/slack` rejects Slack's URL verification

**Severity**: Blocker for BYO in any dev stack not running `apps/signal-gateway`.

When Slack first saves an Event Subscriptions Request URL, it POSTs a `url_verification`
challenge:

```json
{"token":"...","challenge":"...","type":"url_verification"}
```

This payload has no `api_app_id`. `apps/atlasd/routes/signals/platform.ts:47` parsed
`api_app_id` on every POST, so the challenge was rejected with 400 `Missing api_app_id in payload`.

**Log evidence (pre-fix)**:
```
21:18:35.821  Platform webhook received {provider: slack, bodyBytes: 129}
21:18:35.830  slack_signal_missing_app_id {bodyPreview: {"token":"...","challenge":"...","type":"url_verification"}}
21:18:35.831  HTTP POST /signals/slack {status: 400}
```

**Root cause**: `apps/signal-gateway/service/slack_perapp.go:42-51` handles url_verification
locally (echoes the challenge) before forwarding to atlasd. That Go service is not part of
`deno task dev:playground`, so playground-based setups hit atlasd directly and never get a
url_verification handler. BYO's primary use case is local dev where signal-gateway isn't
running, which made this pre-existing bug a hard blocker here.

**Fix applied**:
- `apps/atlasd/routes/signals/platform.ts`: added `SlackUrlVerificationSchema`; at the top of
  the `/slack` route, try-parse it first and return the challenge as `text/plain`. Mirrors
  signal-gateway's behavior.
- `apps/atlasd/routes/signals/platform.test.ts`: +2 cases (url_verification success,
  invalid-JSON 400). 12/12 passing.

**Retest**: Slack Event Subscriptions → **Retry** → Request URL shows green **Verified ✓**.

```
21:29:53.193  Platform webhook received {provider: slack, bodyBytes: 129}
21:29:53.203  slack_url_verification_challenge
21:29:53.204  HTTP POST /signals/slack {status: 200, duration: 5ms}
```

### Finding 2 (P1, upstream-fixed but not yet released): outbound Slack replies fail with `invalid_thread_ts`

**Upstream status**: Already tracked and fixed at `vercel/chat#268` / `vercel/chat#292`.
Issue was filed 2026-04-15, fix merged 2026-04-17. Latest npm publish is
`@chat-adapter/slack@4.26.0` on 2026-04-14 — **pre-merge**, so the fix is on `main` but
hasn't shipped yet. Our QA run rediscovered the bug one day after the upstream fix landed.

**Upstream's root-cause note**: The empty `threadTs` for DMs was intentional — it's used
by `openDM()` subscription matching to route DMs to `onSubscribedMessage` instead of
`onNewMention`. Naïve fixes that change the threadTs source break subscription matching.
The upstream fix **preserves** the empty `threadTs` in the threadId for subscription
semantics and **normalizes** it at method entry for Slack API calls:

```typescript
const { channel, threadTs: rawThreadTs } = this.decodeThreadId(threadId);
const threadTs = rawThreadTs || undefined;
```

**Severity**: Blocks any Slack reply from Friday regardless of credential source (BYO or Link-managed). Pre-existing bug in the outbound path.

**Log evidence**:
```
21:56:53.210  thread_post_failed
  adapterName: slack
  threadId: slack:D0AUE2HRC5Q:
  error: {
    name: Error,
    message: "An API error occurred: invalid_thread_ts",
    stack: at ChatStreamer.stop (chat-stream.js:124)
        → SlackAdapter.stream (@chat-adapter/slack:2959)
        → ThreadImpl.handleStream (chat:1476)
  }
```

The `threadId` is `slack:D0AUE2HRC5Q:` — note the **trailing colon with empty segment**. For a top-level
DM (not a threaded reply), `thread_ts` should be unset, not empty-string. The adapter is
passing `""` to Slack's `chat.update` at stream finalization, which Slack rejects.

**Suspect location**: `@chat-adapter/slack` package's `ChatStreamer.stop` passes the
`thread_ts` slot from the thread ID. If the ID carries `slack:<channel>:<ts>` with missing
`<ts>`, the empty string leaks through.

**Local patch applied to unblock QA** (ephemeral — lives only in `node_modules/.deno/@chat-adapter+slack@4.25.0/.../dist/index.js`, not committed):

1. **Line 1483** — change DM branch to always use a valid timestamp:
   ```diff
   -    const threadTs = isDM ? event.thread_ts || "" : event.thread_ts || event.ts;
   +    const threadTs = event.thread_ts || event.ts;
   ```
2. **Lines 2160, 2188, 2217, 2639, 2854, 2892** — defensive `|| void 0` on `thread_ts` at
   Slack API call sites (mirrors what `postEphemeral` and `uploadFiles` already did). Redundant
   given patch #1 guarantees `threadTs` is never empty, but kept as belt-and-suspenders.

**Caveat vs upstream fix**: this patch differs from vercel/chat#292. Their fix preserves the
empty `threadTs` in the threadId (important for `openDM()` subscription routing) and
normalizes only at Slack API boundaries. Ours overwrites the threadTs at its source. For
Friday's current wiring, this is harmless: `chat-sdk-instance.ts:395-397` routes BOTH
`onNewMention` and `onSubscribedMessage` to the same handler, so the subscription distinction
has no behavioral impact.

**Evidence the patch works**:
```
22:22:17  Agent response: "Hey Sara! Yes, I can reply — what are you testing?"
22:22:17  HTTP POST /signals/slack status: 200 (was 500 before patch)
```
Slack UI shows "1 reply — View thread" (reply is threaded under the user's message because
our patch forces `threadTs` to `event.ts`; acceptable tradeoff for local QA).

**Durability**: patch is ephemeral. `deno cache` re-resolve, `rm -rf node_modules`, or another
machine cloning the repo wipes it. Options when `@chat-adapter/slack@4.27.0` ships:

1. Upgrade and delete the patch (most likely outcome — upstream fix is authoritative).
2. If we need the fix sooner than the release, vendor the package in-tree under a
   `vendor/` directory and point `deno.json` imports at it.
3. Maintain a local patch via `deno.json` scripts that re-apply on `postinstall`.

**Action**: Watch for `@chat-adapter/slack@4.27.0` on npm. Delete local patch once upgraded.

### Finding 3 (P3, documentation): README said "from scratch, not manifest"

**Severity**: Minor doc correction.

`docs/integrations/slack/README.md` (written earlier in this session) steers users toward the
from-scratch UI path and calls the manifest path "the Link-managed path." During QA, I took the
manifest path (hand-pasted a JSON manifest at
`https://api.slack.com/apps/<id>/app-manifest`) because it sets all 15 scopes + event
subscriptions atomically — much faster, equally BYO-compatible.

**Recommendation**: Update the README to present both paths. Mention manifest as a shortcut for
experienced users; keep from-scratch as the default walkthrough.
