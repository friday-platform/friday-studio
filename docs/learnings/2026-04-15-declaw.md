## Eval iteration (post-migration)

After D6 the team-lead loop pivoted to "make the eval suite work end-to-end
and stop being a black box." Three rounds.

### Round 0 — spot check + audit
- `declaw-spotcheck` (wikipedia × 2) confirmed plumbing works but exposed
  the assertion-vs-prompt tension: agent correctly routes static Wikipedia
  to `fetch`, but `assertBrowserFunctional` hard-requires ≥1 `browse` call.
- Audit of `web.eval.ts` post-D6: only the two wikipedia browser cases
  were structurally stale; everything else from the old auth flow had
  already been pulled in D6.
- Sub-agent ported 3 new search cases covering coverage gaps (technical
  lookup / entity disambiguation / hallucination resistance). The
  deleted `research.eval.ts` had nothing the existing search section
  didn't already cover — "port from history" became "design for gaps."

### Round 1 — `declaw-baseline-v2` (commits `49775c5bf` + `ae006f8b9`)
- 12/13 assertion-passed.
- Live logging fix in `tools/evals/lib/context.ts` was the unlock — root
  cause of "evals are silent" was `CapturedLogger`/`CapturedStreamEmitter`
  swallowing every log + stream event into private arrays. Fixing it at
  the harness layer lit up every agent's `logger.info` calls in one shot.
- Real failures hidden under the assertion pass:
  - `failure/no-calendar-access` ran **175 browse + 15 search calls**
    trying to find calendar data instead of refusing. task-complete=0.
  - 5 of 6 search cases scored `has-citations=0` — agent dropped URLs
    from synthesis output.
  - Mixed cases skipped expected tools (`tool-selection=0`) because
    inputs let `fetch` succeed.
  - HN browser case had the same structural issue as wiki — fetch
    succeeds, assertion demands browse.

### Round 2 — `declaw-baseline-v3` (commits `d7dd3abe9` + `c0f2dcd03`)
**13/13 PASS.** Two parallel sub-agents (prompt fixes + case fixes)
landed simultaneously without conflict because they touched
non-overlapping files (`prompts.ts` vs `web.eval.ts`).

Key wins:
- no-calendar: **190 tool calls → 0** (agent now refuses in one step).
  Refusal heuristic uses concrete trigger phrases ("my calendar",
  "my meetings", "my email") rather than abstract "personal data" rules
  — per tuning-agents principle, models treat enumerated examples as
  reliable patterns and abstract rules as loopholes.
- Citations: 1/6 → 5/6 search cases score `has-citations=1.00`. One-line
  prompt rule with inline `(https://source.com)` example was enough.
- Mixed cases: `tool-selection=0` → `1.00` on both. Fix was rewriting
  inputs so fetch alone *cannot* satisfy the task (e.g., "click the
  destination link" forces browse).
- HN browser case: fetch:1 (FAIL) → browse:10 (PASS). Same fix pattern —
  task changed from "title of #1" (static-fetchable) to "username of
  top comment in #1's thread" (requires interactive navigation).

## Observations worth remembering

- **Untracked trees break `git mv`.** The whole `packages/bundled-agents/src/web/`
  subtree was untracked on declaw. Storm's D1 rename discovered that `git mv
  steel-utils.ts agent-browser-utils.ts` fails because the source is not in the
  index. Plain `mv` + `git add` is the path. Worth flagging in CLAUDE.md or
  the committing skill if this recurs.
- **Deletion with no diff is valid.** Po's D3 deleted `credential.ts` but the
  file was untracked, so there was literally no commit to make. The correct
  answer was "no commit needed, delete is on disk." A teammate newer to the
  flow could easily get stuck trying to force a commit here.
- **Module-level preflight doesn't run during `evals list`.** Luka noticed that
  `deno task evals list` is static discovery — the top-of-file
  `execFileAsync("agent-browser", ["--version"])` call never fires during
  listing. Only at actual run time. Useful for designing future preflight gates.
- **`deno task typecheck` includes transitive workspace files.** Every
  teammate in this session reported "my files are clean but the workspace has
  unrelated errors." The review flow accommodated this by checking the diff
  against changed files only, then escalating the workspace-level gate as a
  separate concern. Possible future skill gap: a "typecheck-my-diff" helper
  that scopes to changed files.
- **Snapshot test failure messages should include the actual hash.** Ferox's
  D5 snapshot test emits the computed hash in the `expect().toBe(EXPECTED)`
  failure so the developer can copy-paste the new constant straight from
  output. Worth stealing for other prompt snapshot tests.
- **Heavy-cap rule holds up.** Ferox (D5) and Jinju (D2) each did one heavy
  task and shut down clean. Po doing light-then-medium (D3+D4) also worked —
  mixed-weight rotation is fine as long as the cumulative count respects the
  tighter cap.
- **Eval harness silence is a UX bug, not a feature.** Capturing logs into
  buffers without mirroring to stderr made eval runs feel like 20-minute
  black boxes. One change to `CapturedLogger` and `CapturedStreamEmitter`
  to mirror events to stderr (gated by `EVAL_QUIET=1`) made every agent's
  existing `logger.info` calls visible. Lit up unrelated diagnostic value
  too — the search agent's internal "Analyzing query..." progress was
  always there, just invisible.
- **Concrete trigger phrases beat abstract rules in prompts.** The
  refusal heuristic that fixed no-calendar uses literal strings ("my
  calendar", "my email") not categories ("personal data"). The agent
  pattern-matches phrases directly; abstract category language gets
  interpreted away. This validated a tuning-agents principle in vivo.
- **Parallel sub-agents avoid conflicts via file partitioning.** The two
  round-2 fix agents ran simultaneously and committed cleanly because
  their domains were disjoint (prompt content in `prompts.ts` vs case
  inputs in `web.eval.ts`). When parallel work would touch the same
  file (e.g., prompt refusal AND citations both touching `prompts.ts`),
  combining into one agent is the right call — both fixes share the
  prompt's sha256 hash constant and would race.
- **`failure-behavior` scorer is too strict.** It checks
  `wasFailure === testCase.shouldFail` where `wasFailure = !result.ok`.
  So a graceful refusal (returns `ok` with text "I can't help with
  that") scores 0 even though it's the desired behavior. Splitting
  `shouldFail` into `shouldError` (throws) and `shouldRefuse` (returns
  ok with refusal text) would let the metric reward both correctly.

   graceful refusals get credit.

   that didn't recover. Agent burns 21 tool calls but strips URLs. Worth a
   focused look.

   commands in v3 — task-complete is 0.95 but step-efficiency is 0.12.
   Either accept Algolia is hard, simplify the case, or instrument the
   agent for a bail-out heuristic on long browse loops.
