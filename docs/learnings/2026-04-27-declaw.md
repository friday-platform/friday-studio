# Team lead learnings — 2026-04-27 — declaw branch (ChatSdkNotifier)

## Shared-worktree git hygiene (recurring)

Two teammates independently ran into the same class of incident in wave 1 of
this session:

- One did `git reset --soft HEAD~1` followed by `git restore --staged` to drop
  another teammate's staged file from a commit. The reset succeeded but their
  re-commit never landed — the rebase sequence rolled their work back into the
  working tree, and they marked the task complete believing it was committed.
- The other ran `git stash` to set aside in-flight work, hit lock-file
  conflicts on pop, and recovered via a manual `git checkout stash@{0} -- <file>`
  + `git reset HEAD <files>` sequence.

Both incidents trace to the same root cause: trying to manage the *other*
teammate's staged paths instead of just controlling their own with `git add
<specific-paths>`.

Codify in CLAUDE.md / teammate prompt:

- In shared worktrees, **never** run `git stash`, `git reset --soft`, or
  `git restore --staged` to deal with another teammate's pending changes.
- Always stage only your own files with `git add <specific-paths>` and then
  `git commit`. If `git status` shows another teammate's edits in your staging
  area, just don't include them in your `git add`; leave them alone.
- After committing, verify with `git log -1 --stat` that your files actually
  landed before marking the task complete.

## TeamCreate ordering vs task creation

`TeamCreate` creates a per-team task list. Tasks created **before** `TeamCreate`
land in a different (default/global) task list and are invisible to teammates
spawned into the team. Order: `TeamCreate` → `TaskCreate`s → spawn teammates.

## Task-status drift

A teammate marked Task #1 `completed` after their commit was rolled back, so
status said "done" while the working tree still held uncommitted changes.
Reviewers must verify with `git log -1 --stat` after `completed` notifications,
not just trust the status field.
