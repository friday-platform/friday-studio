# Chat Export — Team Lead Learnings (2026-05-04)

- Team task lists are scoped per-team; the parent's TaskList is invisible to teammates. Need to seed tasks under the team after `TeamCreate`. Cost me one round-trip with Luka who couldn't find T1.
- AI SDK discriminated unions on `msg.parts[]` defeat structural narrowing — type guards via `isRecord` + `typeof part.type === "string"` produce *less* `as`-cast pollution than the original inline implementations. Worth flagging in the karpathy/CLAUDE.md guidance for future render-helper work: prefer this pattern over `(part as { ... }).field`.
- The agent-playground side had `metadata.{startTimestamp,timestamp,endTimestamp}` references at the call site but the `ChatMessage.metadata` type didn't carry those fields until T11 added them. Original timestamp implementation was effectively reading undefined.
- Manual-verify "click button in browser" gates are awkward when the user's local daemon is on a different worktree. Programmatic smoke (direct-import Chat → zip → unzip-and-eyeball) plus pending integration tests (T20) is a fine substitute for a tracer-bullet whose AC says "click and download." Worth pre-deciding in future task ACs: which manual gates can be replaced by smoke, and which need real browsers (e.g. CSS hover states, drag-and-drop).
- Po flagged that `archiver` (per AC) was new dep when `jszip` was already in tree at `artifacts.ts`/`bundle.test.ts`. Approved jszip — established pattern wins over AC literal-spec when the cost is a new dep + inconsistency. Doc design ACs around lib choices should specify *why* the lib was chosen so reviewers/teammates can judge whether the reason still holds.
- **XSS via markdown passthrough (FOLLOW-UP)**: Storm flagged during T19 that `markdownToHTML` (marked, HTML passthrough enabled) doesn't sanitize raw HTML in user/assistant text. Same renderer is used in the live chat UI (`chat-message-list.svelte`) — so the export inherits whatever risk model the live UI has. Worth a separate ticket: should `markdownToHTML` sanitize, should the export pre-sanitize, or is "all chat content is trusted-by-workspace" the actual policy? Right now it's implicit.
- Test-fixture `as unknown as AtlasUIMessage` pattern is unavoidable given the AI SDK's strict discriminated union types. Used in `render.test.ts` (T11) and `export-html.test.ts` (T19). Treating it as a documented test-only escape hatch — sweeping both files if/when the SDK exposes a cleaner fixture API.
- The CLAUDE.md "no `as`" rule needs a documented exception for test fixtures; otherwise reviewers waste cycles re-litigating the same point per task. Worth proposing an addendum.
