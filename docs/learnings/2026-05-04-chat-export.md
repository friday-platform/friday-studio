# Chat Export — Team Lead Learnings (2026-05-04)

- Team task lists are scoped per-team; the parent's TaskList is invisible to teammates. Need to seed tasks under the team after `TeamCreate`. Cost me one round-trip with Luka who couldn't find T1.
- AI SDK discriminated unions on `msg.parts[]` defeat structural narrowing — type guards via `isRecord` + `typeof part.type === "string"` produce *less* `as`-cast pollution than the original inline implementations. Worth flagging in the karpathy/CLAUDE.md guidance for future render-helper work: prefer this pattern over `(part as { ... }).field`.
- The agent-playground side had `metadata.{startTimestamp,timestamp,endTimestamp}` references at the call site but the `ChatMessage.metadata` type didn't carry those fields until T11 added them. Original timestamp implementation was effectively reading undefined.
- Manual-verify "click button in browser" gates are awkward when the user's local daemon is on a different worktree. Programmatic smoke (direct-import Chat → zip → unzip-and-eyeball) plus pending integration tests (T20) is a fine substitute for a tracer-bullet whose AC says "click and download." Worth pre-deciding in future task ACs: which manual gates can be replaced by smoke, and which need real browsers (e.g. CSS hover states, drag-and-drop).
- Po flagged that `archiver` (per AC) was new dep when `jszip` was already in tree at `artifacts.ts`/`bundle.test.ts`. Approved jszip — established pattern wins over AC literal-spec when the cost is a new dep + inconsistency. Doc design ACs around lib choices should specify *why* the lib was chosen so reviewers/teammates can judge whether the reason still holds.
- **XSS via markdown passthrough (FOLLOW-UP)**: Storm flagged during T19 that `markdownToHTML` (marked, HTML passthrough enabled) doesn't sanitize raw HTML in user/assistant text. Same renderer is used in the live chat UI (`chat-message-list.svelte`) — so the export inherits whatever risk model the live UI has. Worth a separate ticket: should `markdownToHTML` sanitize, should the export pre-sanitize, or is "all chat content is trusted-by-workspace" the actual policy? Right now it's implicit.
- Test-fixture `as unknown as AtlasUIMessage` pattern is unavoidable given the AI SDK's strict discriminated union types. Used in `render.test.ts` (T11) and `export-html.test.ts` (T19). Treating it as a documented test-only escape hatch — sweeping both files if/when the SDK exposes a cleaner fixture API.
- The CLAUDE.md "no `as`" rule needs a documented exception for test fixtures; otherwise reviewers waste cycles re-litigating the same point per task. Worth proposing an addendum.

# Round 2 (2026-05-04, second implementing-tasks session)

- **Pre-existing dirty worktree contaminates teammate commits.** Po's T1 commit on `apps/atlasd/routes/workspaces/chat.ts` swept up an unrelated dirty hunk (a 5th `artifactBytesById` arg to `renderChatToHTML` that no overload accepts) because they ran `git add <file>` on a file that already had unrelated WIP. The result type-checked fine on Po's local view (since each teammate runs in isolation) but broke the team's HEAD on integration. Mitigation for future sessions: lead must `git stash` (or commit) any pre-existing dirty state before spawning teammates. Add to the teammate prompt: "before staging, confirm `git diff --staged` matches *only* the lines you wrote — if it doesn't, unstage and use `git add -p`."
- **Playground exempts `console.*` per its CLAUDE.md** (`tools/agent-playground/CLAUDE.md` § Conventions: "console.* is allowed — dev tool exemption"). Lead overstated the "no `console.*`" rule to teammates working under that tree. Future teammate prompts should distinguish daemon code (banned) from playground code (allowed). Daemon is `apps/atlasd/`, `packages/`. Playground exemption only applies under `tools/agent-playground/`.
- **Playground SPA-default is a hidden trap for SSR-dependent features.** `tools/agent-playground/src/routes/+layout.ts` sets `export const ssr = false` for all routes (cited reason: AI SDK transitive `@opentelemetry/api` SSR failures). This means `kit.inlineStyleThreshold: Infinity` is invisible across existing pages. Any new export-style SSR route MUST opt back in explicitly via its own `+page.ts` setting `ssr = true; csr = false`. Worth flagging in the playground CLAUDE.md so the next person doesn't hit it.
- **Hono RPC type inference is occasionally non-deterministic.** Hit a TS2589 "Type instantiation is excessively deep" on `apps/atlasd/routes/workspaces/chat.ts` after the contamination revert; running `deno check` again immediately produced a clean result. No code change between runs. Likely a Deno LSP cache hiccup interacting with Hono's recursive route inference. If it shows up, retry before debugging.
- **Design doc was wrong about a daemon endpoint.** `docs/plans/2026-05-04-chat-export-svelte-render-design.md` § Daemon API surface claims `GET /api/workspaces/:wsId/chat/:chatId/artifacts` "already exists, reused as-is." It doesn't — only `/api/artifacts?chatId=X` exists (mounted at `apps/atlasd/src/atlas-daemon.ts:1066`). Patina caught this on T4 and used the working endpoint with a TODO for T8. Decision: keep using the global `/api/artifacts?chatId=X` endpoint; do NOT add a chat-scoped daemon route just to match the design doc. Reason: avoids touching daemon code that's mostly being deleted in T9; the global endpoint works today; "shapelier" URL was design-doc aspiration not reality. **Worth a separate fix-up commit on the design doc** at session end so future readers don't repeat this discovery.
- **Slugify helper destination revisited.** Initial inclination (mine) was that the slugify rule could "duplicate, document the daemon's copy" per the design doc's both-options wording. Patina's plan went the other way and wanted a shared helper in `packages/core`. Final decision was a **third location**: `tools/agent-playground/src/lib/export/artifact-zip-path.ts`. Reasons (recorded for T8's brief): daemon's copy gets deleted in T9 anyway, packages/core/chat/export/render.ts is for transcript-render helpers (not asset-path computation), and both long-term consumers (T4 preview's `resolveUrl` and T8 orchestrator's zip placement) live in playground. Cross-package shared helpers should be motivated by an actual cross-package consumer, not aspirational reuse.
- **`$lib` alias was missing from the root `vitest.config.ts`.** Pre-existing pattern adds aliases for `$app/environment` etc. but had no `$lib` entry. Patina added it on T4 so SvelteKit-flavoured route tests resolve the same way the dev server does. Worth flagging to anyone writing route-level tests in this monorepo: check `vitest.config.ts` aliases first.
- **Design doc lied about archiver being in deps.** Both this session and the prior one hit the exact same trap: design doc text says "Use `archiver` (already on the dep tree)"; it isn't. Only `jszip` is. Jinju on T8 verified via `grep -rn "archiver"`, `find node_modules`, and `package.json` read. Decision (matching prior session's): use jszip. Recipients see the same bundle the about-to-be-deleted daemon path produced. Streaming concern flagged for hypothetical T18 size profiling. **Process learning**: when a design doc names a specific library, verify it's actually installed before writing code that depends on it. Costs 30 seconds; saves a wasted attempt. Worth a one-line addendum to the design doc to correct the archiver claim.
- **Lead must drain message queue between teammate handoffs.** I missed Jinju's archiver-vs-jszip ask because I was reviewing Patina's T4 message at the time, and the question only surfaced when Jinju sent a follow-up that included the answer. Jinju made the right call on their own (and explicitly cited my own instructions — "if you can't make archiver work, message me"), but the round-trip cost a beat. **Process improvement**: after approving a teammate's task, scan the inbox for any other pending teammate questions before assigning the next task. Saves the "I'll just decide for myself" branch in scenarios where I do actually want a say.
- **Hallucinated commit SHA in peer DM blocked downstream work.** Storm DM'd Ferox at the start of T10 saying T9 was committed at SHA `e2f1c3d` and "branch is yours." That SHA didn't exist (real T9 commit was `96ad711`). Ferox correctly refused to commit T10 on an unverifiable SHA and surfaced the mismatch to the lead. Defensive moves: (1) Teammates should always copy SHAs from `git rev-parse HEAD` immediately after commit, never from memory. (2) Teammates should report task completion to lead first; peer DM is a useful side-channel but lead is the SHA-verifying authority. (3) Receiving teammates should default to refuse-on-unverifiable rather than trust-then-break-HEAD. Worth adding to the teammate prompt: "if a peer claims a commit at a SHA, verify with `git log --oneline -3` before depending on it."
