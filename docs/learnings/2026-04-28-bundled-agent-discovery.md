# Team Lead Learnings — bundled-agent discovery (2026-04-28)

Branch: `feature/bundled-agent-discovery`
Tasks: #30–#38

## Observations

- Test fixtures for nested schema types (e.g., `WorkspaceConfig`) tempt `as unknown as Type` double-casts when the literal is structurally incomplete. Recurring pattern. CLAUDE.md's `as`-prohibition exception list ("JSON.parse in test helpers before immediate Zod parse") does NOT cover fixture builders. The clean fix is `Schema.parse({...})` from `@atlas/config` — validates the fixture and produces a real value. Worth surfacing in CLAUDE.md as guidance for future test authoring.
- `bundledAgentsRegistry` is keyed `Record<string, entry>` where alias keys (`browser`, `research`) point at the same object as `web` (referential equality, not duplication). Iterating `Object.entries(registry)` yields aliases; iterating the canonical `bundledAgents` array does not. Non-obvious; comes up any time a tool tries to enumerate bundled agents.
- ai-sdk `tool({ execute })` return type doesn't propagate to the call site — test code sees `execute()` output as `unknown`. The clean way to narrow without `as` is to export a result Zod schema and `Schema.parse(raw)` in the test runner. (Ponderosa added `ListCapabilitiesResultSchema` for this.)
- Branch is shared with the validation/hallucination workstream (#20-#29). `git log -1` and `HEAD~1` can land on a parallel-team commit, not your teammate's. Always inspect the specific hash from the teammate's report.
- The design doc and #33's task description both contained an internal contradiction: § Module Boundaries said "runtime is unchanged"; § Testing Decisions listed an `unknown_bundled_agent` structural-issue test that requires a new runtime validation. Patina caught it — escalating before papering over saved a scope explosion. Whenever a "description-only" task ships an AC bullet that requires touching runtime code, that's a contradiction worth surfacing.
- Shared-worktree race: when wave-2 teammates ran in parallel (#31, #33, #34), one teammate's `git add` swept in another's staged changes. The recovery cascade — `git reset HEAD~1` to clean up — silently dropped *both* a sibling teammate's work-in-flight AND a downstream teammate's already-committed work. Net result: two completed commits dropped from history, work surviving only in the working tree. The fix was for the team lead to re-stage and recommit each piece. Mitigation for future runs: teammates working in the same worktree should `git add <specific-files>` (already in CLAUDE.md), and avoid resets when something looks tangled — escalate to the lead instead. A worktree-per-teammate setup would prevent this entirely.
- Concrete mitigation Patina discovered mid-wave: `git commit -o <paths> -m "..."` (the `--only` flag) commits exactly the listed paths regardless of what else is staged — bypasses the shared index entirely. Cleaner than `git add path && git commit` in a parallel worktree because the index can be mutated between those two calls. Worth folding into CLAUDE.md's "Shared worktrees" note alongside the existing `git add <specific-files>` rule.
- Subtle agent-config gotchas Jerry surfaced while authoring `references/agent-types.md` (worth folding into CLAUDE.md's gotchas section): (1) `AtlasAgentConfigSchema.env` is workspace-level *overrides* only — bundled agents actually read from `process.env` at runtime. The `requiresConfig` array returned by `list_capabilities` is the source of truth for what the agent needs, not the `env:` field on the workspace config. (2) `LLMAgentConfigSchema` has no top-level `env` field — env-dependent settings go in `config.provider_options`. A workspace YAML showing `env:` under a `type: llm` agent will parse silently but be functionally inert at runtime. The `WorkspaceAgent` runtime view only extracts `env` for `atlas` and `user` types (see `packages/config/src/mutations/workspace-agents.ts:62-97`).
- Discovery-tool bypass under strong description signal: Almond's #38 eval surfaced that the LLM picks `type: "atlas", agent: "web"` 4/4 *without* calling `list_capabilities` first. The rewritten `upsert_agent` description (which enumerates atlas/user/llm shapes inline with concrete bundled-agent examples) plus the workspace-api skill instruction together carry enough signal that the discovery tool is short-circuited. Not a regression — the gate condition (bundled-agent-chosen + no-redundant-mcp-enable) is satisfied 4/4. But it's a signal that `list_capabilities` will only become load-bearing when (a) new bundled agents ship that the description doesn't enumerate, or (b) MCP-only intent dominates. Worth tracking in post-ship telemetry: if a new bundled agent ships and adoption lags despite being in the registry, the SKILL's "call `list_capabilities` at session start" instruction may need stronger framing.
- Eval-runner harness pattern for chat-flow tests Almond surfaced: workspace-chat's real handler is too coupled to be invoked headlessly (daemon HTTP client, ChatStorage, MCP discovery). For behavioral assertions on LLM judgment, build a synthesized harness that drives `streamText` directly with the *actual* `prompt.txt` + `SKILL.md` content loaded from the live tree, plus verbatim tool descriptions. Tests the same judgment without the wiring overhead, and the source-of-truth-loaded approach means the eval auto-tracks SKILL/description rewrites.
