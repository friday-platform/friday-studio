## Observations worth remembering

- **Teams have their own task list.** Tasks created by the lead BEFORE `TeamCreate` live on the lead's personal list and are invisible to teammates. Recreate tasks AFTER creating the team, or teammates will report "task not found." Caught this on the first spawn — Luka saw an empty TaskList until I recreated all three tasks.
- **Idle notifications are noise, not signal.** Teammates go idle after every turn. The skill doc says this explicitly, but it's easy to read 2–3 consecutive idle pings as "stuck" and send a nudge that's not needed. Only act on idle if a task is stalled relative to a known deadline.
- **Plan approval gate is worth it for the first teammate even on medium tasks.** I gated Luka's Task #1 (Tracer Bullet) and Storm's Task #2 (first long-lived WebSocket adapter, new primitive) — both plans already reflected the critical gotchas I'd surfaced, which meant less rework. For Task #3 (docs) I skipped the gate and it shipped clean anyway.
- **Storm proactively hardened auth error detection without being re-prompted.** After my first approval they saw my feedback note about structured `.code` / `.status` detection and self-shipped a follow-up commit + test before shutdown. Teammates will voluntarily address "nice-to-have" feedback if it's concrete.
- **Pre-loading opensrc adapter source into context paid off massively.** Every teammate referenced specific line numbers (`index.ts:1638`, `:1683-1692`, `:1724`) in their plans and commits. Without the `npx opensrc @chat-adapter/discord` fetch earlier in the conversation, each teammate would have burned ~5–10 minutes re-doing that research.

## CLAUDE.md / docs candidates

- **`chat_sdk_instance_created` now carries a `discordGateway: boolean` field** — worth documenting alongside the existing list of `adapters:` in the log line, so future adapter additions follow the "one log line proves full wiring" pattern.
- **`@chat-adapter/discord` constructor throws `ValidationError` synchronously on any missing cred** — general pattern lesson: adapter factories that throw in constructors must be gatekept by resolvers returning `null` on incomplete creds. Worth adding as a one-liner under the "Chat adapter credential resolution" CLAUDE.md gotcha.
- **`WebhookOptions.waitUntil` is typed `(task: Promise<unknown>) => void`** (contravariant) — declaring the task with `Promise<void>` at the caller site fails typecheck even though the runtime value works. Catch this in the adapter section if a gotchas list exists there.
- **Deno lint `require-await` directive placement** must sit on the line DIRECTLY ABOVE the `async` arrow — placing it above the `vi.fn(...)` wrapper produces "Ignore for code was not used." Storm hit this; worth a one-liner in the Deno gotchas list.

## Process observations

- **Heavy/medium/light task sizing held up.** Tasks #1 (multi-file impl) and #2 (new primitive + lifecycle) were clearly heavy → 1 task per teammate. Task #3 was medium → 1 teammate also handled it alone. Rotating teammates after heavy tasks kept context windows fresh; both Luka and Storm's second commit (the follow-up) came in under the cap because follow-ups don't count.
- **Sub-agent second-opinion review found real bugs in the PLAN, not the code.** Before any implementation, spinning up a general-purpose agent to review the Discord plan against the adapter source caught three issues that would have shipped broken: missing `chat.initialize()` step, `waitUntil` promise-awaiting subtlety, and resolver not enforcing all-or-nothing. Cost ~2 minutes, saved an entire fix wave.
