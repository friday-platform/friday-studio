# Team Lead Learnings — Discord BYO (declaw, 2026-04-22)

## Session summary

Three-task wave: Discord BYO messaging integration via `@chat-adapter/discord` in legacy Gateway mode. Commits: `ac2205ae` (adapter + env resolver) → `d38af6de` (supervisor) → `d61a63b8` (structured auth error detection) → `1a8d7d56` (docs + QA plan).

## Observations worth remembering

- **Teams have their own task list.** Tasks created by the lead BEFORE `TeamCreate` live on the lead's personal list and are invisible to teammates. Recreate tasks AFTER creating the team, or teammates will report "task not found." Caught this on the first spawn — Luka saw an empty TaskList until I recreated all three tasks.
- **Idle notifications are noise, not signal.** Teammates go idle after every turn. The skill doc says this explicitly, but it's easy to read 2–3 consecutive idle pings as "stuck" and send a nudge that's not needed. Only act on idle if a task is stalled relative to a known deadline.
- **Plan approval gate is worth it for the first teammate even on medium tasks.** I gated Luka's Task #1 (Tracer Bullet) and Storm's Task #2 (first long-lived WebSocket adapter, new primitive) — both plans already reflected the critical gotchas I'd surfaced, which meant less rework. For Task #3 (docs) I skipped the gate and it shipped clean anyway.
- **Storm proactively hardened auth error detection without being re-prompted.** After my first approval they saw my feedback note about structured `.code` / `.status` detection and self-shipped a follow-up commit + test before shutdown. Teammates will voluntarily address "nice-to-have" feedback if it's concrete.
- **Pre-loading opensrc adapter source into context paid off massively.** Every teammate referenced specific line numbers (`index.ts:1638`, `:1683-1692`, `:1724`) in their plans and commits. Without the `npx opensrc @chat-adapter/discord` fetch earlier in the conversation, each teammate would have burned ~5–10 minutes re-doing that research.

## CLAUDE.md / docs candidates

- **`chat_sdk_instance_created` now carries a `discordGateway: boolean` field** — worth documenting alongside the existing list of `adapters:` in the log line, so future adapter additions follow the "one log line proves full wiring" pattern.
- **`@chat-adapter/discord` constructor throws `ValidationError` synchronously on any missing cred** — general pattern lesson: adapter factories that throw in constructors must be gatekept by resolvers returning `null` on incomplete creds. Worth adding as a one-liner under the "Chat adapter credential resolution" CLAUDE.md gotcha.
- **`WebhookOptions.waitUntil` is typed `(task: Promise<unknown>) => void`** (contravariant) — declaring the task with `Promise<void>` at the caller site fails typecheck even though the runtime value works. Catch this in the adapter section if a gotchas list exists there.
- **Deno lint `require-await` directive placement** must sit on the line DIRECTLY ABOVE the `async` arrow — placing it above the `vi.fn(...)` wrapper produces "Ignore for code was not used." Storm hit this; worth a one-liner in the Deno gotchas list.

## Process observations

- **Heavy/medium/light task sizing held up.** Tasks #1 (multi-file impl) and #2 (new primitive + lifecycle) were clearly heavy → 1 task per teammate. Task #3 was medium → 1 teammate also handled it alone. Rotating teammates after heavy tasks kept context windows fresh; both Luka and Storm's second commit (the follow-up) came in under the cap because follow-ups don't count.
- **Sub-agent second-opinion review found real bugs in the PLAN, not the code.** Before any implementation, spinning up a general-purpose agent to review the Discord plan against the adapter source caught three issues that would have shipped broken: missing `chat.initialize()` step, `waitUntil` promise-awaiting subtlety, and resolver not enforcing all-or-nothing. Cost ~2 minutes, saved an entire fix wave.

# Session 2: Delegate tool implementation

## Session summary

9-task wave to replace `do_task` with `delegate` (in-process nested `streamText`) in workspace-chat. Plan: `docs/plans/2026-04-17-delegate-tool-design.v4.md`.

## Observations worth remembering

- **The "no `as`" rule has no test-helper exception, but generic SDK boundaries make it unavoidable.** Ferox's Task #8 test helper accessing `tool({...})`'s `execute` member needed `as never` to recover the call signature from AI SDK's opaque generic. After `"execute" in toolObj` narrowing, the first two casts collapse, but converting `Function`-typed result to a typed call signature still requires either a cast or a wrapper. The wrapper pattern `(input, options) => fn(input, options)` skips the final cast. Worth adding as a CLAUDE.md note: when test-helper boundaries hit opaque SDK generics, prefer wrappers over casts.
- **Result-type field names: `data` / `error.reason`, not `output` / `message`.** Task #8's spec referenced `payload.output` and `payload.message`, but the actual `result.ts` exports `ok({data})` and `err({reason})`. Ferox correctly read the source over the spec — this is the right behavior. Worth noting that task descriptions copied from design docs can drift from the actual API; teammates should always verify against source.
- **Adding subpath exports to avoid pulling FFI barrels into tests is a recurring pattern.** Ferox added `@atlas/core/streaming` to skip dragging `@db/sqlite` into the vitest graph for test-only imports. Same pattern as the existing `@atlas/core/session/types` carve-out. CLAUDE.md already mentions this for the web-client; could generalize to "vitest tests must use subpath exports for any `@atlas/core` import".
- **Anthropic rejects Zod's `discriminatedUnion` JSON Schema for tool inputs.** Jinju discovered this on Task #2 manual verification: Zod produces `{oneOf: [...]}` at the top level with no `type` field, which Anthropic rejects with `tools.<n>.custom.input_schema.type: Field required`. The fix is `jsonSchema<T>({type: "object", properties: {...}})` (Anthropic-friendly flat shape) plus a runtime parser that re-validates against the strict discriminated union. CLAUDE.md already has the broader rule "tool() requires jsonSchema() for inputSchema, not z.object() for parameters" — but the specific failure mode for `z.discriminatedUnion` is silent: the child stream emits a single `error` chunk and `result.text` later throws `AI_NoOutputGeneratedError`. Worth a one-liner in the AI SDK gotchas list.
- **`result.text` rejects with `AI_NoOutputGeneratedError` when the stream is tool-call-only.** Jinju's Task #2 fix: pair `await result.steps` and `await result.text` in separate try blocks so `toolsUsed` (computed from steps) survives a text rejection. Otherwise a tool-call-only delegate run loses its outline ledger. Also worth a CLAUDE.md note under AI SDK gotchas.
- **Walking `step.content[]` is the cleanest way to compute per-call outcome.** Jinju found that `step.toolResults[]` is success-only and `step.toolCalls[]` is call-only — neither tells you "did this call error". `step.content[]` is the union of `tool-call | tool-result | tool-error` parts; one walk produces the full picture. Useful pattern for any future code that needs outcome reconstruction from streamText results.
- **Production-code `as` casts at union-spread boundaries are sometimes unavoidable.** Jinju's `proxy-writer.ts` does `{...chunk, toolCallId: ...} as AtlasUIMessageChunk` because spreading a discriminated union into an object literal loses the discriminator narrowing. Per-branch handling would balloon the code 5x. The cast is structurally safe (we're rebuilding the same chunk), but the rule has no carve-out for this. Worth either documenting an exception in CLAUDE.md or recommending a typed helper pattern for "modify one field of a union".
- **`ReadableStream.tee()` is the right primitive for "observe + forward" patterns where one consumer is async/external and the other is in-process.** Almond's Task #5 used `uiStream.tee()` to split the child's UIMessage stream into an observer branch (timing capture, in-process) and a parent branch (proxy → SSE → external). Tee buffers per branch with no cross-branch back-pressure, so neither consumer blocks the other. Useful pattern beyond delegate — anywhere you need to instrument a stream without slowing the primary consumer.
- **AI SDK v6 `result.steps` triggers `consumeStream()` internally via its own teeStream branch.** Almond discovered this: `await result.steps` resolves whether or not `toUIMessageStream()` is consumed downstream. Means an in-process observer can safely drain its own tee branch without waiting on the parent's SSE sink, and `await result.steps` won't deadlock on either branch. Worth a CLAUDE.md note under AI SDK gotchas.
- **Steps are authoritative; chunks contribute timing only.** Almond's design split: `name` / `input` / `stepIndex` / `outcome` / `summary` come from `result.steps`, `durationMs` comes from chunk arrival timestamps. Reason: chunks can be replayed/reordered/duplicated on stream restart, but steps are final-form. Generally useful pattern when reconciling streamed and resolved data.
- **`JSON.stringify(undefined)` returns `undefined`, not the string `"undefined"`.** Same for functions and Symbols. Must guard with `?? "[unserializable]"` (or similar) when the value source is `unknown`. Almond's `truncateForLedger` does this. Worth a CLAUDE.md note under TypeScript gotchas.
- **Anthropic's tool input schema requires `type: "object"` at the root.** Bare `{oneOf: [...]}` / `{anyOf: [...]}` (which Zod's `discriminatedUnion` produces via `z.toJSONSchema`) is rejected with `tools.<n>.custom.input_schema.type: Field required`. Leela hit this on Task #9 with the deterministic `gh`/`bb`/`jira` agents whose `OperationConfigSchema` is a discriminated union. The fix in `createAgentTool`: detect non-object roots via `z.toJSONSchema(schema).type !== "object"` and fall back to a default `{prompt: string}` schema. Their handlers parse JSON from a free-form prompt anyway, so structurally safe. Generalize: any agent that passes through to `createAgentTool` and has a non-object root schema needs the fallback. Worth adding to CLAUDE.md AI SDK gotchas, alongside Jinju's `discriminatedUnion` discovery from Task #2.
- **Vitest does NOT honor the `with { type: "text" }` import attribute.** `import SYSTEM_PROMPT from "./prompt.txt" with { type: "text" }` resolves to the path string, not the file content, in vitest. Existing tests that compare prompt to itself still pass (both sides are the path string). Content-level assertions in tests need `await readFile(promptPath, "utf-8")` from `node:fs/promises` instead. Worth a CLAUDE.md vitest gotcha entry.
- **Top-level imports in agent module files can fail at module-evaluation time before any test runs.** Leela found that `bundledAgents.map((agent) => [agent.metadata.id, ...])` (or any top-level statement that touches a transitively-mocked dependency via `importActual`) throws synchronously at module load, masking the actual test target. Fix: `vi.mock("@atlas/bundled-agents", () => ({ bundledAgents: [] }))` in handler tests. Pattern: when adding a new top-level import to an agent module, audit downstream tests for module-graph coupling.
- **`UIMessageStreamWriter.merge` is fire-and-forget — synchronous writes after `merge` race ahead of merged chunks.** Patina hit this on Task #6: emitting a synthetic `delegate-end` terminator via `writer.write()` after `proxy.merge(stream)` produced terminator-before-children ordering. Fix: drain the stream chunk-by-chunk via `proxy.write()` in a `for await` loop, then await completion before emitting the terminator. The `merge` API is for async fan-in where ordering doesn't matter; sync ordering requires manual drain. Worth a CLAUDE.md AI SDK gotcha entry.
- **`ReadableStream.tee()` decouples backpressure between branches but does NOT decouple completion.** When the source ends, both branches end. Patina relied on this: `await observerDone` AND `await forwardDone` together pin the moment when every child chunk has been processed by both sides. Generally useful pattern when you need a "fence" across multiple consumers of the same stream.
- **`TransformStream.flush` callback fires when the SOURCE closes, not when the consumer closes.** Patina used this for active-merge counters in the proxy writer's `merging` state — `activeMerges--` in `flush` correctly tracks "this merge has finished draining" regardless of whether the parent writer has consumed the wrapped output yet.
- **`streamText` does not reliably throw on AbortSignal.aborted — it can simply truncate the stream cleanly.** Patina's Task #6 pattern: detect abort in `finally` via `abortSignal?.aborted` (not in `catch`) since the abort path doesn't always raise. Worth a CLAUDE.md AI SDK gotcha — this affects any code that wraps streamText with abort-aware cleanup.
- **Chromium `<details>` does NOT apply UA `display: none` to non-summary children when closed.** Storm hit this on Task #4: the classic pattern of "set `display: flex` when `[open]` and let the browser hide otherwise" leaves children visible after auto-collapse. `<details>` uses shadow-DOM slotting that light-DOM children bypass. Must explicitly set `display: none` baseline + `display: flex` (or `block`) on `[open]`. Worth a CLAUDE.md note in the SvelteKit / browser gotchas section. Generally relevant for any custom collapsible markup using `<details>`.
- **Design gap discovered on Task #7: AI SDK v6's `validateUIMessages` strips top-level non-schema fields from messages.** The v4 plan presumes `msg.state === "done"` reaches the client on chat reload so the reducer's crash fallback can promote stuck delegate children to `output-error`. Reality: `UIMessage` only types `id`, `role`, `metadata`, `parts`. A persisted `state: "done"` does NOT survive `validateUIMessages` on the server's `GET /chat/:id`. Practical impact: the `parent.state === "done"` rule is dormant on the persistence path unless something stamps `state` after the fetch. Ponderosa's surgical fix in `user-chat.svelte` stamps `state: "done"` on assistant messages during rehydration (a message loaded from disk is by definition past its live turn) — but the architectural question remains: is rehydrate-time stamping the right home for this signal, or should it live in `metadata` / a part / a different signal entirely? Follow-up task created.
- **Vite + SvelteKit dev server resolves source modules via `/@fs/<absolute-path>` AND `/src/<rooted-path>.js`** — useful for invoking module functions in the browser console for ad-hoc integration tests when the UI flow is hard to reach. Ponderosa used `await import('/src/lib/components/chat/extract-tool-calls.js')` to call the Task #7 reducer directly with a hand-built fixture, avoiding the round-trip through persistence + `validateUIMessages` (which would have stripped the field under test). Same code path the prod component uses. Useful pattern for any future "unit-test this pure function in its real module graph" check.
- **The playground stores `chatId` per workspace in `localStorage["playground:lastChatId:<wsId>"]`.** Flipping that key + reloading is the supported way to point at a synthetic chat file written directly to `~/.atlas/chats/<wsId>/<chatId>.json`. The `?chatId=...` URL query param is NOT honored — the `user-chat.svelte` rehydration reads from localStorage exclusively. Useful for crash-fixture-style integration testing.
