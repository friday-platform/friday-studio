# Review: declaw — memory_narrative alias removal

**Date:** 2026-04-23
**Branch:** declaw
**Verdict:** Needs Work

## Summary

Removes deprecated `memory_narrative_{append,read,forget}` tool aliases across the MCP server, FSM allowlist, platform SDK, and agent-tool filters; implements the previously stubbed `MdNarrativeCorpus.forget()`; and rewrites workspace-chat's memory toolset to add general-purpose `memory_save/read/remove` tools targeting arbitrary `workspaceId/memoryName` pairs. The removal work is clean and consistent. The new workspace-chat toolset has one critical issue (authorization bypass) and three correctness gaps to address before merge.

## Critical

### Workspace-chat memory tools bypass `resolveCorpus` — arbitrary cross-workspace writes

**Location:** `packages/system/agents/workspace-chat/tools/memory-save.ts:39,70,87`

**Problem:** The MCP-path `memory_save` calls `resolveCorpus` which validates the requested `workspaceId`/`memoryName` pair against `memory.own` / `rw` mounts in `workspace.yml`, rejecting undeclared corpora. The new workspace-chat tool skips this entirely — it takes raw `workspaceId` and `memoryName` straight from the LLM and posts to `/api/memory/{workspaceId}/narrative/{memoryName}`. The daemon memory POST handler adds no ownership validation.

**Evidence:** `mcp-server/src/tools/memory/save.ts:43-50` calls `resolveCorpus` and returns an explicit error for undeclared names. `apps/atlasd/routes/memory/index.ts` POST handler constructs `MdNarrativeCorpus` from path params with no auth check. A prompt-injected conversation can write to any workspace ID on the host, including the kernel workspace.

**Recommendation:** Before hitting the daemon, call `resolveCorpus` with `op: "write"` (same as the MCP path), or add workspace membership validation in the daemon POST handler. The declaration gate from `resolveNarrativeCorpus` was not replaced.

**Worth doing: Yes** — real regression introduced by this PR, not pre-existing.

## Important

### All three tools hardcode `/narrative/` regardless of corpus strategy

**Location:** `packages/system/agents/workspace-chat/tools/memory-save.ts:39,70,87`

**Problem:** All URLs are built as `/api/memory/{workspaceId}/narrative/{memoryName}`. If a workspace declares a corpus with `strategy: kv`, `dedup`, or `retrieval`, the tools write to a separate `narrative/` directory the declared corpus never reads from. The tool returns `{ saved: true }`; no error surfaces. The old MCP path (`mcp-server/src/tools/memory/save.ts:56-59`) returns an error for non-narrative strategies.

**Recommendation:** Either (a) consult `resolveCorpus` to get the strategy and use the correct route, or (b) add to the tool description that it only targets `strategy: narrative` corpora and validate at call time. The MCP path already does this; the workspace-chat path silently diverges.

**Worth doing: Yes** — silent wrong-location write on non-narrative corpora.

---

### `_workspaceId` parameter is dead code

**Location:** `packages/system/agents/workspace-chat/tools/memory-save.ts:28`

**Problem:** The call site (`workspace-chat.agent.ts`) still passes `workspaceId`, but the function now ignores it. CLAUDE.md says to remove dead code entirely, not silence it with `_`. If the intent is for the LLM to target arbitrary workspaces, remove the param and update the call site. If the intent is to default to the session workspace, restore the constraint.

**Worth doing: Yes** — dead code, violates CLAUDE.md rule; also the call site passing data that goes nowhere obscures whether the workspace constraint is still in effect.

---

### `forget()` MEMORY.md filter matches on substring, not parsed ID

**Location:** `packages/adapters-md/src/md-narrative-corpus.ts:106`

**Problem:** The JSONL path is correct: it parses each line as JSON and does `parsed.id === id` (exact comparison). The MEMORY.md path uses `line.includes(\`(id: ${id})\`)` — a substring match. If an entry's text field contains `(id: <target-id>)` verbatim (quoted note, user paste), that line is silently deleted. JSONL and MEMORY.md diverge.

**Evidence:** `parseMemoryMd()` at line 130 already uses a regex that captures the `id` group — this pattern is available in the class for exact matching.

**Recommendation:** Reuse the existing line-parsing regex from `parseMemoryMd()`, capture the `id` group, and compare against `id` exactly.

**Worth doing: Yes** — real correctness issue, asymmetric with the JSONL path.

## Tests

### `forget()` does not cover MEMORY.md cleanup

**Location:** `apps/atlasd/routes/memory/get.test.ts:146`

The DELETE test writes entries directly to `entries.jsonl`, bypassing `corpus.append()`, so MEMORY.md is never populated and the `forget()` MEMORY.md code path is never exercised. Seed via `corpus.append()` instead, then assert both `entries.jsonl` and `MEMORY.md` are cleaned up after DELETE.

---

### `memory_read` and `memory_remove` missing network failure tests

**Location:** `packages/system/agents/workspace-chat/tools/memory-save.test.ts`

`memory_save` tests both HTTP failure and network failure. `memory_read` only tests the happy path; `memory_remove` tests HTTP failure but not network failure. The `catch` branches in both are unexercised. Add `mockRejectedValueOnce(new Error("ECONNREFUSED"))` tests matching the existing `memory_save` pattern.

---

### `PLATFORM_TOOL_NAMES` lists `memory_save/read/remove` but `createPlatformTools` doesn't include them

**Location:** `packages/agent-sdk/src/platform-tools.ts:84-91`, `platform-tools.test.ts:351-370`

These tools come from `workspace-chat/tools/memory-save.ts` (direct injection) and `mcp-server/src/tools/memory/` (MCP registration), not from `createPlatformTools`. The test correctly omits them from the `createPlatformTools` integration check. The gap is documentation: the comment on `PLATFORM_TOOL_NAMES` says "canonical registration is in `mcp-server/src/tools/index.ts`" but doesn't note that `memory_save/read/remove` are registered there (confirm `registerMemorySaveTool`, `registerMemoryReadTool`, `registerMemoryRemoveTool` are present in `index.ts` — they are). Add a comment to the test noting that workspace-chat tools use a separate direct-injection path.

## Needs Decision

1. **`_workspaceId` removal scope:** Dropping the parameter means the chat agent can write to any workspace the LLM names. Is that the intended UX — or should the tool default the `workspaceId` to the current session's workspace when the LLM omits it? The current state (param accepted but ignored) is the worst of both worlds.

2. **`resolveCorpus` for workspace-chat:** Adding the resolve step adds a network round-trip to the daemon for every memory operation. An alternative is to gate at the daemon route level (add membership validation there), which centralizes auth. Worth deciding which layer owns this before fixing issue #1.
