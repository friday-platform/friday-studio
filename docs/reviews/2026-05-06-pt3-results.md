## Architectural decisions baked in

- **Pt1's supervisor flip stays.** Job-tool returns `{artifactIds, summary}`. Pt3 confirmed structural correctness via H4 (67.5% reduction on a lean fixture; magnitude is workload-dependent — pt1's −95.1% holds on heavier fixtures).
- **Pt2's validation-as-skill stays.** External via `generateObject` was reversed in pt3 K2 — judge now routes through delegate proper, gaining Phase 8 budgets + Phase 11 provenance + on-demand `artifacts_get`. The runner-side wire-up is a follow-up.
- **`validating-llm-outputs` is the canonical validation skill.** Auto-detect classifier returns `skip | self`; never `external`. K6 added per-MCP overrides + `run_code: { readOnly }` opt-in over the regex defaults — author intent wins.
- **Activity UI is the surface for Phase 12 elicitations** (closes pt1's Phase 12.D). G4's sweeper makes the page UX honest; H5's coverage protects the SSE handler's teardown logic.
- **Crystallization stays out of scope.** K5 explicitly defers `successSignal` rather than removing the field — keeping it forward-compatible costs nothing.
- **Activity → FSM resume not yet wired.** Decision D-pt3-1: document the "answer does not resume — re-trigger the job" limitation in (a) the `request_tool_access` tool description and (b) the G1 Activity page UX. Suspend/resume implementation deferred to pt4 / a separate plan (needs FSM persistence + KV consumer wiring on `ELICITATION_STATUS` + cascade re-entry path).
- **N2 defensive fallback over architectural redesign.** Free-form + self path injects `record_validation`; some LLMs end on that tool call instead of emitting text alongside it. Rather than skipping injection (which would break B6's guard), recover the earlier text from `steps[]`. Minimally invasive; B6 contract preserved.
- **K4 consumed-on-read over event emission.** The cascade-dispatch path is the sole consumer of `completedSessionMetadata`; consumed-on-read GC inside `getSessionJobResult` is simpler and more robust than emitting a `session:consumed` event with a downstream listener.

## Learnings (operator-facing)

- **Sub-agent commit-step skipping is a recurring failure mode.** Under heavy parallelism, multiple sub-agents drafted code, ran (or claimed to run) the triplet, then *stopped short of `git commit`*. Commits only landed when (a) the parent intervened with a dedicated commit sub-agent, or (b) the agent re-checked progress mid-loop and corrected. **Mitigation:** brief sub-agents with an explicit "do not return until `git log -1` shows your commit" gate. Treat any "I will commit" / "I should commit" language without a follow-up `git commit` invocation as a failure to deliver.
- **`lint-staged` stash/restore races corrupt commit attribution under heavy parallelism.** During the J2/J3/L5 dispatch wave, three concurrent agents committed back-to-back and `lint-staged` (which moves unstaged files aside, runs format/lint on staged files, then restores) absorbed sibling agents' staged files into the wrong commits. Result: commit `c4eb85f` is mis-labeled "J2" but actually contains J3's content; J2's real session-adapter changes landed separately at `f937aca` via a follow-up commit sub-agent. **Mitigation:** for any future parallel batch where multiple agents stage files in the same window, either (a) serialize commits (one agent waits for `git status` clean before committing), or (b) use `isolation: "worktree"` for true file isolation. The pt2 race-learning was about same-file edits; this is a new layer about commit-time stash collisions.
- **Concurrent sub-agents on adjacent files silently clobber edits.** G4 reported intermediate Edits to `jetstream-adapter.ts` / `types.ts` / `mod.ts` were silently overwritten and had to be re-applied via `Write`. The worktree had multiple `claude` processes hitting overlapping packages. **Mitigation:** confirm not just disjoint *target files* but also disjoint *modify-lock surfaces* — a test in package X may pull in module-level state from package Y that another agent is mutating.
- **Harness flakes are real but rare.** First all-phases run on E1.1's HEAD returned 5/8 with three failures sharing a single "SSE event walk returned empty" signature; re-run on the same HEAD came in 8/8. Diagnosed as daemon-side SSE timing, not a regression. **Mitigation:** if a harness regression's failures cluster on a single signature AND the runtime delta is logically excluded from the failing scenarios, re-run before assuming regression.
- **Reviewer premise can be outdated.** H6's brief asserted `composeValidationBlock` "had no isolation test." It actually had 6 — the true gap was the try/catch arm. Sub-agents should ground in the test file's current state before coding the brief literally; report back if the brief's premise doesn't match reality.
- **The −95.1% headline is workload-dependent.** Production had ~9.4 KB multi-doc legacy output; QA fixture emits a single ≤200-word summary. H4 measured 67.5% on the lean fixture. The structural flip is correct; the headline reduction depends on inner-job output size. Future benchmarks should document the fixture profile alongside the percentage.
- **Sub-agents took shortcuts on architectural directives.** Pt2's B7 used `generateObject` directly instead of routing through delegate proper (the original spec). Pt3 K2 was the cleanup. Pt3 K1 reversed direction mid-flight when the operator clarified that scratchpad was not supported — the original "ship MCP tools" spec became "remove the surface entirely."
