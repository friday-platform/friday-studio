## Findings — new notes

Not regressions; worth knowing.

**F1. `POST /:ns/:name` enforces lint errors, unlike `/install`.**
- Attempted to simulate a UI Save by crafting a publish payload via
  `jq` and curl. Publish returned `400 Skill failed lint` with
  `frontmatter-schema` + `description-missing` errors because the
  payload only carried `{description, instructions}` and the linter
  re-validated frontmatter that doesn't include them.
- This is by design (direct API callers must supply valid frontmatter),
  but the plan should call out that the publish route and the install
  route have different gates. Added to the v2 §3.6 route audit as a
  note for next pass.

**F2. Fork response body parses cleanly; earlier "invalid control
char" traced to my own malformed payload.**
- Embedded literal newlines in an `instructions` field written through
  curl landed in the DB unescaped and broke downstream parsers. The
  daemon accepted the POST. Two options to harden: reject on input
  (strict JSON re-encode before persist), or sanitize on output
  (ensure JSON-encode escapes all control chars). Low priority — only
  reproducible by a client sending malformed JSON — but logged.
