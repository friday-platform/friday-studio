## Systemic Root Cause

**14 of 20+ Sentry issues shared one architectural pattern:**
catch blocks at HTTP route boundaries used string pattern matching or generic catches instead of `instanceof` discrimination. Errors that are semantically 4xx (user/config errors) fell through to 500 catch-alls, creating misleading alerts.

---

## Remaining Architectural Gaps (Future Work — No Active Sentry Issues)

1. **FSM Engine generic catch** (`fsm-engine.ts:1454`) — generic catch wrapping entire action execution; no `instanceof` discrimination
2. **String-based error matching** in 4 workspace routes (`POST /`, `GET /:id`, `GET /:id/config`, `DELETE /:id`) — `errorMessage.includes("not found")` instead of `instanceof`
3. **Source maps not uploaded to Sentry** (ATLAS-WEB-CLIENT-A4 deferred) — minified stack traces in web-client errors
4. **Two monitoring alerts blocked** — `session-failure-spike.yaml` and `reconciliation-errors.yaml` reference non-existent Prometheus metrics
